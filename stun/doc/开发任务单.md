# STUN 开发任务单

本文档将现有设计文档中的目标拆解为可执行开发任务，并给出对应代码位置和验收标准，作为后续实现与联调的执行依据。

## 1. 目标范围

本任务单覆盖以下内容：

- 多机器协作 STUN 服务的控制面和数据面
- NTRS Hub 与 NTRS Node 的成员管理
- MQTT Broker 接入、遗嘱消息、retain 消息
- 双机最小部署和联调验证
- NAT 探测、协作探测、风险分类

## 2. 现有代码基线

当前仓库已经具备以下基础能力：

- STUN 编解码基础
- MQTT 客户端封装
- NTRS 消息编解码
- `ntrs_hub`、`ntrs_node`、`ntrs_peer` 示例入口
- Hub 集群状态模块 `ntrs_hub_state`
- 控制面认证与会话模块 `ntrs_auth`
- 基础单元测试入口 `stun_unit_tests`

结合最近一轮代码整理，基线还新增了以下进展：

- NTRS 外层帧已经稳定为自定义二进制帧加 TLV 结构，不再需要额外引入 FlatBuffers、protobuf 或 msgpack-c。
- `ntrs_codec` 已支持 `BOOL`、`U16`、`U32`、`I32`、`U64` 等 typed 字段，端口、计数、布尔值、RTT、版本号等高频字段已改为二进制编码。
- `ntrs_peer`、`ntrs_node` 示例中主要数值字段已切换到 typed getter/setter，不再依赖字符串格式化和反解析。
- Hub 状态机已从示例主流程中拆分为独立模块，已经实现 `boot_id` 代际替换、旧 LWT 忽略、按心跳间隔的过期摘除。
- 控制面已引入第一版共享密钥加短期 session token 认证流程，`AUTH_REQ/RSP`、`REGISTER_REQ`、`HEARTBEAT_REQ`、`UNREGISTER_REQ`、`SESSION_CREATE_REQ` 等路径已接入校验。
- `ntrs_nat_detect` 与 `ntrs_peer` 已收敛到同一套完整异步 NAT 探测流程，顺序为 `stun1 -> change-port -> change-ip -> stun2`。
- `ntrs_node` 已支持 `CHANGE-REQUEST` 相关 full STUN 协同行为：本地 `3479/udp` 响应 `change-port`，协同 Node 的 `3478/udp` 代发 `change-ip`。
- 会话协商已支持交换 `local_ip:local_port`，并向 `peer` 下发 `host_local / srflx_primary / srflx_secondary` 候选；同机或同局域网场景可优先命中 `host_local`。
- 协议热点路径读写都已大幅切换到 `FieldTag + typed TLV`，并通过 `tag_index` 做 O(1) 查找，字符串比较和文本数值回退已明显减少。
- 当前代码已经重新完成一次完整构建验证，`ntrs_hub`、`ntrs_node`、`ntrs_peer` 与 `stun_unit_tests` 可正常编译。

后续工作不是从零实现，而是围绕现有骨架补齐协议、状态机、协作逻辑和部署流程。

## 2.1 当前进度更新

按当前仓库状态，可以把任务推进度粗分为三类：

- 已完成第一轮落地：协议二进制化、typed TLV 基础能力、主要示例调用切换、Hub 状态机第一版、控制面认证第一版、编译验证。
- 已有实现但未完全收敛：snapshot 语义冻结、认证强度、会话令牌边界、事务 ID 统一、测试覆盖深度。
- 尚未形成交付闭环：双机部署文档、Broker 权限边界、回归测试矩阵和联调脚本。

当前判断：

- 阶段 A 中的“消息编码形式”已经基本收敛，但“语义收敛”和“字段冻结”还没完全结束，尤其是 `boot_id`、`cluster_version`、`assignment_version`、`presence/heartbeat` 的状态语义还需要继续固定。
- 阶段 B 已经不再只是骨架，Hub 状态机和控制面认证已经有了第一版实现，Node 也已合并控制面与 STUN 服务能力，但仍缺少更严格的恢复语义、权限边界和安全约束。
- Node 当前主循环仍是 `select + 阻塞 recv/send + 同步跨节点请求` 的单线程模型，慢连接和慢联邦请求会直接阻塞控制面、STUN 响应和 MQTT 心跳处理；full STUN 探测虽然已异步化，但节点间 federation 仍是按请求临时建 TCP，没有收敛成长期连接管理器。
- 阶段 C 的主路径已可运行并完成一轮 full STUN 收敛，但联邦连接复用、更多复杂 NAT 组合验证和更严格的探测边界仍未完全收敛。
- 阶段 D 已经有单元测试起点，但覆盖范围仍偏窄，目前主要覆盖 typed TLV 编解码、Hub 状态机和认证作用域，尚未覆盖慢连接阻塞、控制会话续期、联邦探测权限边界等高风险路径。

## 4. 推荐实施顺序

1. 先消除 Node 单线程阻塞点，避免单个慢连接或慢联邦请求拖死整个控制面和 STUN 服务。
2. 再修控制会话续期语义，把 `AUTH_REQ` 生成的控制 token 生命周期和 `REGISTER/HEARTBEAT` 租约对齐。
3. 然后收紧协作探测权限边界，避免 `FILTER_PROBE_REQ` 和跨节点代发探测被滥用为任意 UDP 发送能力。
4. 再继续补 Hub 恢复语义、snapshot/assignment 冻结和 STUN 事务 ID 统一。
5. 最后扩回归测试和部署脚本，把上述高风险路径做成可重复验证的闭环。

补充说明：

- “协议编码改造”这一小步已经先行完成，后续顺序不再是“是否改二进制”，而是“基于现有 typed TLV 收敛语义、状态机和测试”。
- 因此当前推荐顺序里，真正的下一步焦点已经从“编码形式”转移到“运行时不会卡死、会话不会自然失效、协作探测不会越权发送”这三类现实风险。

## 5. 基于当前代码的优先修复清单

这一节不是新增需求，而是基于当前仓库现状，对最该优先补齐的问题做代码审查式排序，便于直接排期。

### P1. 消除 `ntrs_node` 的单线程阻塞点

问题：

- `ntrs_node` 主循环虽然用了 `select`，但实际读包仍通过阻塞式 `recv_exact()` 拉完整帧；只要某个 TCP 客户端只发半包，整个循环就会卡住，STUN、MQTT 和其他客户端都被一起拖住。
- `NAT_PROBE_REQ`、`FILTER_PROBE_REQ` 在同一个事件循环里同步调用远端 Node，请求链上每个连接都可能阻塞 1 秒以上，多个候选节点串行尝试时会把单线程循环长时间挂死。
- 当前风险不属于“性能优化”，而是明确的可用性问题，线上只要出现慢链路、探测超时或恶意半包，就会把节点拖成假死。

优先原因：

- 这是当前最直接的单点失效源。
- 不先处理阻塞模型，后续再补 Hub、认证或探测策略，实际运行时仍会因为一个慢连接把整条路径拖死。

建议落点：

- [stun/examples/ntrs_node.cc](stun/examples/ntrs_node.cc)

验收补充：

- 半包、慢包、远端超时不会阻塞本地 STUN 响应。
- 跨节点探测失败时不会拖住其他控制请求和 MQTT 心跳。
- 节点在恶意或异常客户端存在时仍能持续处理其他连接。

### P2. 修正控制会话 TTL 与心跳租约脱节

问题：

- `AUTH_REQ` 返回的控制 token 默认只活 30 秒，但 `REGISTER_REQ/HEARTBEAT_REQ` 成功后并不会续期控制会话。
- Peer 示例注册后每 5 秒发一次心跳，看起来像长连接常驻模式，但服务端会在 30 秒后把控制会话扫掉，随后心跳和注销都会自然变成 `AUTH_REQUIRED`。
- 当前租约语义分成“peer lease 30 秒”和“control session 30 秒”两套并行计时器，但两者没有刷新关系，运行时间一长就会自相矛盾。

优先原因：

- 这是稳定场景下就会触发的真实功能错误，不需要攻击流量也会出现。
- 如果不先统一会话续期语义，后续所有联调结论都会建立在“运行 30 秒内没问题”的假象上。

建议落点：

- [stun/examples/ntrs_node.cc](stun/examples/ntrs_node.cc)
- [stun/src/ntrs_auth.cpp](stun/src/ntrs_auth.cpp)
- [stun/examples/ntrs_peer.cc](stun/examples/ntrs_peer.cc)
- [stun/src/ntrs_client.cpp](stun/src/ntrs_client.cpp)

验收补充：

- 长时间心跳不会因为控制 token 自然过期而中断。
- `REGISTER_REQ`、`HEARTBEAT_REQ`、`UNREGISTER_REQ` 对同一条连接的续期规则一致。
- 控制连接断开、重连、过期后的 token 生命周期可预测且可测试。

### P3. 收紧协作探测与跨节点代发探测的权限边界

问题：

- 当前 `FILTER_PROBE_REQ` 和 `SERVER_SEND_PROBE_REQ` 允许调用方向任意 `target_ip:target_port` 发送 UDP 探测包，服务端没有把请求目标限制在当前会话、当前协作节点或当前会话令牌的授权范围内。
- 这意味着只要拿到共享密钥并完成 `AUTH_REQ`，就可以把 Node 以及它联动的其他 Node 当成任意目标的 UDP 发送器。
- 当前默认开发密钥仍是固定值，风险不是理论上的，而是“默认部署就可被误用”的级别。

优先原因：

- 这是明确的安全边界缺失。
- 这类能力一旦公开可达，就不只是协议错误，而是会把节点变成外发 UDP 流量放大点和探测工具。

建议落点：

- [stun/examples/ntrs_node.cc](stun/examples/ntrs_node.cc)
- [stun/src/ntrs_auth.cpp](stun/src/ntrs_auth.cpp)
- [stun/include/ntrs_auth.h](stun/include/ntrs_auth.h)

验收补充：

- 只有被授权的会话才能触发过滤探测和跨节点代发探测。
- 请求目标、可用节点和会话作用域之间存在可验证绑定关系。
- 默认配置下不能把 Node 用作任意地址的 UDP 发送器。

### P4. 修正联邦端点解析只支持 IPv4 字面量的问题

问题：

- Node 与 Peer 多处对 `host:port` 做了解析，但真正发起 TCP/UDP 连接时仍直接走 `inet_pton(AF_INET, host)`，这只支持 IPv4 字面量。
- 一旦 assignment、配置文件或 CLI 里填的是域名，解析表面上会通过，真正联邦探测时却会静默失败。
- 当前实现也不支持 IPv6 端点，和“endpoint 是 host:port”这套接口表述并不一致。

优先原因：

- 这会直接影响双机部署与跨节点探测的可用性。
- 不先修这个问题，后续很多联调失败会被误判为 Broker、路由或 NAT 问题。

建议落点：

- [stun/examples/ntrs_node.cc](stun/examples/ntrs_node.cc)
- [stun/examples/ntrs_peer.cc](stun/examples/ntrs_peer.cc)
- [stun/src/ntrs_client.cpp](stun/src/ntrs_client.cpp)

验收补充：

- 域名、IPv4、后续需要时的 IPv6 端点语义明确。
- assignment 下发的控制端点和本地 CLI 端点都能按相同规则解析并连接。

### P5. 把现有回归测试扩到运行时高风险路径

问题：

- 当前测试已经覆盖 typed TLV、Hub 状态机和认证基本作用域，但仍然没有覆盖慢连接阻塞、控制会话续期、跨节点探测授权和域名端点解析这几类最容易在联调中踩中的问题。
- 另外示例工程当前没有显式 Linux-only 编译门禁，平台约束还停留在代码事实层，没有固化到构建层和 CI 层。

优先原因：

- 前四项如果没有自动化验证，修完一次之后很容易被后续重构重新带回来。
- 这些问题都不是“看日志大概没事”能覆盖的，必须补确定性的回归用例。

建议落点：

- [stun/test/ntrs_regression_test.cc](stun/test/ntrs_regression_test.cc)
- [stun/test/CMakeLists.txt](stun/test/CMakeLists.txt)
- [stun/examples/CMakeLists.txt](stun/examples/CMakeLists.txt)

验收补充：

- 至少新增慢连接/半包不拖死主循环、控制 token 续期、跨节点探测权限约束、域名端点解析四类回归用例。
- 示例工程的 Linux-only 约束要么写进 CMake，要么通过文档和 CI 明确拒绝非 Linux 构建。

## 6. 下一步计划

下一阶段不建议再做大范围协议改造，而是把现有第一版实现补成“稳定运行且边界明确”的闭环。按实际依赖关系，建议直接按下面顺序推进：

1. 先拆掉 `ntrs_node` 主循环里的阻塞点。
   目标是把 TCP 控制读写、跨节点探测请求和 MQTT 驱动从“单线程阻塞链路”里拆开，至少保证半包、慢连接和联邦超时不会拖死节点。

2. 再统一控制 token 和 peer lease 的续期语义。
   目标是把 `AUTH_REQ` 得到的控制会话、`REGISTER/HEARTBEAT` 维持的在线租约、`SESSION_CREATE` 派生的 peer token 三者边界讲清楚，并让长连接在正常心跳下不会自然掉鉴权。

3. 然后收紧协作探测的授权范围。
   目标是让 `FILTER_PROBE_REQ`、`SERVER_SEND_PROBE_REQ` 只能在受控的会话和受控的目标范围内执行，默认部署下不能被当成任意 UDP 发送接口。

4. 再补 Hub 恢复语义、端点解析和事务 ID 统一。
   目标是继续冻结 `snapshot/event/assignment` 语义，修掉 hostname 端点无法真正连接的问题，并把 STUN 探测规则补齐。

5. 最后扩最小回归测试集。
   目标不是追求大而全，而是优先补上阻塞主循环、控制会话续期、探测授权边界、端点解析、Hub 恢复语义这些当前最容易回退的高风险路径。
