## 1.2 Types 字段定义 (1 字节)

| 值 | 名称 | 描述 | TODO |
|-------|-------|-------|-----------|
| 0x01 | Initial | ClientHello 建连标志, 携带Version帧, 如果用户选择加密则携带Crypto帧 | None |
| 0x02 | Handshake | ServerHello 携带确认信息 | 0-RTT同样响应, 同时携带kFrameSessionToken帧更新会话密钥 |
| 0x03 | HandshakeDown | 携带HandshakeDown帧, 同时可携带数据 | 用于服务器确认路径可达 |
| 0x04 | 0-RTT | 向服务重新发送数据, 如切换网络等. 携带会话票据密钥 | 需要在保活时间内完成, 否则服务器销毁资源需要重新建立连接 |
| 0x05 | ConnectionClose | 相对端发送ConnectionClose帧 | 对端响应 ConnectionClose 并等待 3 * PTO(有可能对端未收到, 故需要等待), 超时后销毁资源 |

### 1.3.1、优缺点及优化
#### 优点
> - 更低的建连延迟
> - 状态少, 实现简单
> - 与 handshake_timeout 结合可控
>   用一个明确的超时窗口回收半连接状态，可以：
>   - 限制资源占用时间
>   - 让“半连接”不是无限悬挂
#### 缺点
> - DoS面更大
> 2-way handshake 最大的问题是：服务端在发出 Handshake 后就进入 half-open 并保留状态。攻击者可以伪造大量
> Initial（源地址可伪造的UDP），让你产生海量 HandshakeSent 状态，直到超时才回收。
> - 地址不可达/源地址伪造时, 会白发很多 Handshake
>   因为服务端会对每个 Initial 回 Handshake：
>   - 攻击者用受害者IP作为源地址 → 会使得服务器把 Handshake 打到受害者
>   - 如果 Handshake 比 Initial 大, 就形成放大

#### 优化点
> - 引入 "无状态地址验证"
>   做法类似 QUIC Retry / DTLS HelloVerify:
>   - 服务端首次收到 Initial 不分配状态, 而是回一个带 cookie/token 的 Handshake
>   - 客户端下次带 token 再来, 服务端才进入 HandshakeSent 或直接 Connected
>   故增加`HandshakeDown`帧, 在第一次发送数据时携带 (客户端收到Handshake响应时进入HandshakeReceived状态)
>   `解决 源地址伪造 (spoofing) 导致的 状态耗尽`
> - 半连接阶段严格 anti-amplification
>   在服务端尚未确认客户端地址可达(address validated)之前, 服务端在该连接上允许发送的数据量必须被严格限制,
>   避免被用作 UDP 反射/放大攻击, 也避免被伪造源地址的 Initial flood 拖垮. (QUIC 的经典做法是 3x 放大限制)
>   - 明确 "地址已验证" 的定义
>     收到客户端的HandshakeDone帧并检验 cookie/token 通过
>   - anti-amplification
>     bytes_in: 从该五元组/连接ID收到的 UDP payload 字节数累计
>     bytes_out: 向该五元组/连接ID发送的 UDP payload 字节数累计
>     约束: bytes_out ≤ 3 * bytes_in
>     超过就禁止继续发送任何新数据 (包括重传、ACK、应用数据、甚至 Handshake 的重复发送)
>   `解决 源地址伪造 (spoofing) 导致的 反射放大攻击`

#### 3.1.1、流ID设计
- 有序
- 支持单向、双向
- 不会复用已关闭流ID

> 流ID的形式是16位无符号整型, 生成流ID时要保障该流ID在连接内是唯一的, 并且打开一个新流时是不能复用已关闭的流ID的.
> 流ID的最低两位与流的类型有关:
> 最低位标识了流的发起方, 值为0表示该流由客户端发起, 值为1表示该流由服务端发起
> 倒数第二位标识流的方向, 值为0表示这是一条双向流, 值为1表示该流是单向流

| 最低有效位 | 第二低有效位 | 值   | 发起方 | 流方向 |
|:--------:|:----------:|:----:|:-----:|:-----:|
| 0 | 0 | 0x0 | 客户端 | 双向 |
| 1 | 0 | 0x1 | 服务端 | 双向 |
| 0 | 1 | 0x2 | 客户端 | 单向 |
| 1 | 1 | 0x3 | 服务端 | 单向 |

#### 3.1.2 问题解释

##### 3.1.2.1、流ID为什么要连接内唯一且不复用
> 1、避免迟到报文与新流混淆
> UTP跑在UDP之上, 网络就会出现以下情况
> - 旧包延迟很久才到(乱序、重传副本、网络重排)
> - 路径迁移导致的更大乱序 (Wifi -> 4G)
> - ACK丢失导致发送方重复发送STREAM帧 (即使对端已关闭流)
>
> 如果流ID可复用, 就会出现以下场景
> 1. 流 10 已完成并关闭
> 2. 一段时间后, 你复用流 10 作为新请求的流
> 3. 网络里迟到的旧STREAM(流10, offset=...) 或 RESET_STREAM 等控制帧到达
> 4. 接收端无法判断该帧属于旧的流10还是新的流10
>
> 结果要么:
> - 把旧数据注入到新请求 (严重语义错误/潜在安全问题)
> - 错误地关闭/重置新流
> - 流量控制/偏移量检查出错导致协议错误
>
> 如果要复用必须需要额外维度来描述当前流的`代`数
> 如此需要增加协议字段、增加双方状态、增加错误处理分支、增加实现复杂度和互操作风险

##### 3.1.2.2、流ID耗尽如何处理
> 使用32位作为流ID, 在可控范围内基本不会耗尽, 但仍需要考虑耗尽后行为
> 1、流ID耗尽后不在支持创建, 已有流ID不影响使用
> 2、尝试创建新流时应响应错误, 流ID耗尽

##### 3.1.2.3、流个数达到限制如何处理
> 当创建的流个数等于对端限制数再次创建流时触发
> 此时需要要等待, 有可能对端资源受限不响应`MaxStreams`帧, 此时触发创建流超时错误
> 建议: 注册调用`registerStreamCanCreate`注册流可创建事件, 当有可创建流时会回调, 类似epoll水平触发

### 3.2 Ack帧
`重传时不重传Ack帧, 下一个包的Ack会携带上一个包Ack帧要确认的包号`
``` c++
struct AckRange {
    uint32_t    gap{0};
    uint32_t    length{0};
};

struct {
    uint8_t         frame_type;
    uint8_t         range_count{0};
    uint16_t        ack_delay{0};  // 实际延迟 = ack_delay << EXP(3), 时间缩放单位是8微秒
    uint32_t        first_ack_range;
    uint64_t        ack_largest{0};
    AckRange*       ack_ranges;
}

// 场景: 接收端收到包号 95-100, 90-92, 80-85
// QUIC 由于存在变长编码, 所以在设计时进行减1处理, 在边界时节省1字节, 这样在连续确认时, 可以节省更多字节. 此处未采用变长编码, 故使用实际数值.
// 这样做的目的也是因为在发生丢包后重传新的PN, 旧序号丢弃, 故在确认机制上本就不连续, 比如丢失的86号包后续可能是101号包或更大.
// 包号状态:
//   80   81   82  83  84  85  86  87  88  89  90  91  92   93  94  95  96  97   98  99  100
//   ✓    ✓    ✓   ✓   ✓   ✓  ✗   ✗   ✗  ✗   ✓   ✓   ✓   ✗   ✗   ✓   ✓   ✓   ✓   ✓    ✓
//
// ACK 帧构造: 
//
//  Largest Acknowledged = 100
//
//  First ACK Range = 6
//  └── 确认 [95, 100] 共 6 个包
//
//  ACK Range Count = 2
//  └── 有 2 个额外的 ACK Range
//
//  ACK Range 1:
//  ├── Gap = 2
//  │   └── 未确认 [93, 94] 共 2 个包 (Gap)
//  └── ACK Range Length = 3
//      └── 确认 [90, 92] 共 3 个包 (Length)
//
//  ACK Range 2:
//  ├── Gap = 4
//  │   └── 未确认 [86, 89] 共 4 个包 (Gap)
//  └── ACK Range Length = 6
//      └── 确认 [80, 85] 共 6 个包 (Length)
//

Ack的设计需要考虑到丢包后更新包号, 这样旧包号就不在会在接收端出现, 所以Ack是不连续的. 按此想法需要维护一个已接收包号集合
struct Range { uint64_t low, high; };
struct RecvHistory {
    std::vector<Range> ranges; // 按 high 降序
};
或
struct RecvHistory {
    std::map<uint64_t, uint64_t> ranges; // 按 high 降序 high <--> low, 从low到high是连续的
};
可以将此理解为一个快照, 当有新包到达时, 更新快照, 并且在Ack阈值范围内生成Ack帧.
这个阈值可以是超时, 可以是到达包号的数量, 比如收到包1后没有新包到达, 那么经过一个rtt就生成Ack帧, 如果包达到的过快, 连续收到5个包, 那么也生成一个Ack帧.
这样Ack帧会携带之前的包号, 如此当重传带有Ack帧的包时就无法再次携带Ack帧, 因为下一个Ack帧已经包含

快照不能一直增长, 故需要有一个阈值去控制, 比如只保留最近窗口内的包号, 设定一个精细窗口: Window = 4096,
更新时
1、largest_acked = ranges[0].high
2、计算最早需要精确追踪的包号: cutoff = largest_acked - Window (注意下限)
3、对所有ranges做清理
    如果某个 range 完全小于 cutoff (range.high < cutoff), 直接删除；
    如果某个 range 部分跨越 cutoff:
        可以裁剪为 [cutoff, range.high], 舍弃更早部分;

取一个MTU 1280 为最大Ack帧存储单元, 得到size = 1280(mtu) - 40(IPv6) - 8(udp) - 20 (utp) - 16(Ack Hdr Size) = 1196
每个AckRange占8字节, 故 range_size = 1196(size) / 8 = 149.5≈149
```

#### 3.4.1、为什么要等 3 * PTO 超时
> ConnectionClose 包会丢失, 没有确认机制, 主动发送的一端不知道对端有没有收到,
> 发送ConnectionClose后直接释放资源会导致对端一直发送Ping帧请求响应, 并伴随着重传, 浪费对端资源

```  c++
struct {
```

### 3.6 ResetStream 帧

```  c++
struct {
    uint8_t     frame_type;
    uint16_t    error_code;
    uint32_t    stream_id;
    uint64_t    final_offset;
}
```
> ResetStream 帧用于主动结束流, 并通知对端此处出现错误, 如用户中断传输
> stream_id 告知重置哪个流
> error_code 通知重置原因
> final_offset 通知最终传输的数据
> - 如果对端实际上已经收到的字节数 > final_size, 这是协议错误 (说明状态不一致/数据乱套)
> - 如果对端收到了部分数据且 < final_size，则知道 "缺的不会再来", 可以终止重组等待并上报应用错误
> 当对端Ack后释放资源

> 注意
> `正常流结束应通过FIN标志, 本地调用close(stream)后会组装带有FIN标志的stream帧, 如果没有数据可发那么stream_data_length是0,`
> `当收到对端也发送的FIN后, 彻底释放. close会保证之前所有数据已安全到达, 但是后续对端发送的数据不会在触发读事件`
> `ResetStream 属于强制中断读写, 先前未确认的数据会丢失`

### 3.7 StreamsBlocked 帧
```  c++
struct {
    uint8_t frame_type;
    uint8_t stream_type;
}
```
> 握手时会传递 TransportParams, 告知对端流(uni/bidi)最大并发数
> 当本端尝试创建一个新流时发现没有额度, 不能创建新流而是发送 StreamsBlocked 帧, 通知对端想创建更多的流
> stream_type 是方向标志, 可以同时告知对端哪个方向上的流被限制了

> 对端收到 StreamsBlocked 帧后
> 非必须增加额度, 受资源策略控制, 如果愿意解除, 需要发送 MaxStreams 帧

> 当本地察觉到对端IP或端口修改后发送 PathChallenge 帧, 用于校验是否一样

### 4、为什么一开始不直接写大点，而是要后续扩容，比如MAX_STREAMS，本身就只是一个数值记录，又不会真实创建这么多流

这是个很好的问题，其实涉及到 QUIC 设计里“保守起步、按需放宽”的整体理念，而不只是 MAX_STREAMS 一个字段。

可以从几个角度看：**资源控制 / 拒绝服务防护 / 协议演进 / 实现复杂度**。

---

## 1. 控制对端“潜在消耗的资源”，防御滥用/攻击

虽然你说：

> MAX_STREAMS 只是一个数值记录，又不会真实创建这么多流。

但从协议语义上讲：

- 当你告诉对端 `max_streams_bidi = N` 时，你是在**承诺**：
  > 只要对端想开到 N 条流，我就有义务接受（协议层面不能拒绝）。

在实现里，这意味着你要**预留足够的能力**来处理这么多潜在的流：

- 流对象（state、buffer、定时器、流量控制状态等）；
- 对应的应用层上下文（例如 HTTP/3 的请求上下文）；
- 决策逻辑（优先级、调度等）；

如果一上来你就给了一个很大的值，比如 10 万条 stream：

- 合法, 对端可能从来不会用这么多；
- 但攻击者可以立刻开到接近上限，让你在短时间内为大量流分配状态 / 结构体 / 队列等；
- 即便“没真的收发很多数据”，大规模流的创建/管理开销本身就可能让服务端崩溃。

**所以初始值小一点，是一种“流控粒度”的安全措施：**  
- 先画一个小圈：比如让对端最多开 100 个流；
- 如果对端行为正常、需求确实多，再发 `MAX_STREAMS` 扩容到 1000、2000；
- 对端每扩一阶流量/并发，都要“花时间”，攻击的坡度就被拉长了，你更有机会检测、限流、ban 掉恶意行为。

---

## 2. 延迟绑定资源，按需分配，避免浪费

即使对端是“好人”，你一上来给个非常夸张的 MAX_STREAMS：

- 你的实现 **要么**：
  - 为潜在流预留足够的数据结构（哈希表/数组预分配、ID 空间管理、锁等），增加内存占用；
- **要么**：
  - 不预分配，但每次创建新流时，都要考虑“ID 空间巨大”的情况，导致逻辑更复杂、慢路径更多。

而用“初始值 + 动态扩容”的设计：

- 初始时只需要支持一个**比较保守**的并发流数量；
- 运行时看需求才逐步扩展；
- 没有需求时，就不会为高上限付出任何代价。

这和内存池/线程池的设计很像：

- 一开始池子不必开特别大，  
- 运行过程中如果负载上来，再逐步增长。

---

## 3. 与 RTT/流控结合：提供“节奏感”，让发送端不能一下子冲满

QUIC 不只是有 MAX_STREAMS，还有：

- `initial_max_data` + `MAX_DATA`；
- `initial_max_stream_data_*` + `MAX_STREAM_DATA`。

这些都是**接收端施加在发送端身上的“节奏控制”**：

- 如果一上来所有上限都特别大，发送端在一个 RTT 内就可以疯狂把窗口填满；
- 在真实网络里，这会导致：
  - 突然的大量队列堆积；
  - 丢包、重传、RTT 抖动；
  - 拥塞控制算法需要在更极端的区间收敛。

而“初始小 + 动态放大”的好处是：

- 在连接刚建立、RTT 估计/路径信息都不充分时：  
  先小心翼翼地发一点；
- 随着 ACK 和路径信息逐渐完善，再扩大窗口、提高并发流数量。

**流控和拥塞控制的一个核心原则就是“慢启动、慎扩容”**，MAX_STREAMS 只是其中的一个维度。

---

## 4. 安全设计的通用哲学：最小权限原则

从安全视角看：

> 不要在一开始就给对端“它可能永远用不到的超大权限/额度”。

QUIC 设计里很多东西都遵循这个原则：

- CID 数量上限（`active_connection_id_limit`）；
- ACK 相关参数（ack_delay 的约束）；
- Stream 数量、数据量上限（`initial_max_*` + `MAX_*`）。

这些值都可以看成是“你愿意暴露给对端使用的资源额度”。  
一开始给少一点，再根据连接行为逐步增加，是**风险可控**的方案。

假设没有这种机制，只有“初始 transport_params 一次性给死”：

- 那你要么一开始就给得很保守，一路都不变 → 对高吞吐应用不友好；
- 要么一开始就给得很大 → 直接把拒绝服务的杠杆交给对端。

“带扩容的初始配置”则兼顾了：

- 安全起步（初值保守）；  
- 性能上限（可以持续放宽上限）；  
- 协议的弹性（随时可根据监控/策略调整扩容步伐）。

---

## 5. 为什么不能“只一个大值，然后实际只创建少量流”

你问的重点在于：

> 反正 MAX_STREAMS 只是一个数，不是真正创建那么多流，为什么还要那么谨慎？

关键不在“当前已经创建多少”，而在于你给对端的**权利承诺**：

- QUIC 协议语义是：只要你（发送方）开到 <= MAX_STREAMS 个流，我就不能在协议层说“拒绝这个流”；
- 这意味着如果对端在一个 RTT 里把新流开到上限，你就**必须要在实现里想办法支撑得住**：
  - 为这些新流建立状态；
  - 给它们一定 buffer；
  - 至少在协议层不能拒绝接收这些流的帧。

这其实就是“系安全带”的问题：

- 你可以说“我车能跑 300km/h，但默认限速 80，慢慢加速”，  
- 也可以一上来就松掉限速，让别人随便踩到 300 —— 显然后者风险大得多。

---

## 6. 实际工程上的好处：协议演进与实现简化

还有一些不是那么直观，但工程上很有意义的点：

1. **协议演进**  
   将“初始设置”和“运行中调整”机制解耦，协议可以：

   - 追加新的帧类型，改变扩容策略；
   - 在不破坏基础语义的前提下调整默认初始值。

2. **实现复杂度更低**  
   对实现者来说：

   - 逻辑是“初始值 = 某个 safe 的默认 + 对端提供的 initial_max_*”；  
   - 再在收到 `MAX_*` 帧时“单调递增”；
   - 不需要在各种地方特别判断“这是不是那种一次性巨大的初始上限”。

---

## 小结

你可以这样记：

- transport_params 里的 `initial_max_*` / `initial_max_streams_*`：  
  是 **第一次给对端的“资源信用额度”**，通常会比较保守；
- `MAX_STREAMS` / `MAX_DATA` / `MAX_STREAM_DATA`：  
  是在**连接运行过程中，逐步提高这个额度**，配合流控和拥塞控制，按需放宽。

之所以不“一上来就写很大”，是因为：

1. 这是**拒绝服务防护**和资源控制的需要；
2. 是**慢启动 / 按需扩容**的流控+拥塞控制设计；
3. 符合“最小权限原则”，降低风险；
4. 让实现和协议演进更灵活。

# lsquic 重传模式 (Retransmission Mode) 详解

这四个枚举值代表 **QUIC 协议中的四种不同的丢包恢复机制**,每种模式有不同的触发条件和重传策略。

---

## 🎯 四种重传模式

### 1️⃣ **RETX_MODE_HANDSHAKE** (握手重传)

#### 含义
握手期间的**快速重传模式**,专门用于加速连接建立。

#### 触发条件
```c
static enum retx_mode
get_retx_mode (const lsquic_send_ctl_t *ctl)
{
    // ✅ 优先级最高
    if (!(ctl->sc_conn_pub->lconn->cn_flags & LSCONN_HANDSHAKE_DONE)
                    && have_unacked_handshake_packets(ctl))
        return RETX_MODE_HANDSHAKE;  // 🔥 握手未完成 + 有未确认握手包
    // ...
}
```

**条件:**
- 握手尚未完成 (`LSCONN_HANDSHAKE_DONE` 未设置)
- 存在未确认的握手包 (Initial/Handshake 包号空间)

#### 超时计算
```c
// From [draft-ietf-quic-recovery-01]:
// alarm_duration = max(1.5 * smoothed_rtt, 10ms) << handshake_count
delay = lsquic_rtt_stats_get_srtt(&ctl->sc_conn_pub->rtt_stats);
if (delay) {
    delay += delay / 2;  // 1.5 × SRTT
    if (10000 > delay)
        delay = 10000;   // 最小 10ms
} else {
    delay = 150000;      // 无 RTT 估计时默认 150ms
}
delay <<= ctl->sc_n_hsk;  // 🔥 指数退避 (每次翻倍)
++ctl->sc_n_hsk;
```

**特点:**
- **超时短**: 1.5 × SRTT (最小 10ms)
- **指数退避**: 每次超时时间翻倍
- **不触发拥塞控制**: 握手期间不减小拥塞窗口

#### 示例时间线
```
T=0ms:    发送 Initial 包 #1
T=10ms:   超时 → 重传 Initial 包 #2
T=30ms:   超时 → 重传 Initial 包 #3 (10ms × 2)
T=70ms:   超时 → 重传 Initial 包 #4 (10ms × 4)
T=150ms:  收到 ACK → 握手完成
```

---

### 2️⃣ **RETX_MODE_LOSS** (丢包重传)

#### 含义
基于 **丢包检测** 的重传,是最常见的重传模式。

#### 触发条件
```c
if (ctl->sc_loss_to)
    return RETX_MODE_LOSS;  // 🔥 有丢包超时时间设置
```

**条件:**
- `sc_loss_to` (loss timeout) 非零
- 表示已检测到丢包,需要在特定时间重传

#### 丢包检测逻辑
```c
static int send_ctl_detect_losses(...) {
    for (packet_out in unacked_packets) {
        // 🔥 条件 1: 包号差距过大 (Packet Threshold)
        if (largest_acked - packet_out->po_packno > PACKET_THRESHOLD) {
            mark_as_lost(ctl, packet_out);
        }
        
        // 🔥 条件 2: 超时未确认 (Time Threshold)
        if (now - packet_out->po_sent > loss_delay) {
            mark_as_lost(ctl, packet_out);
        }
    }
}
```

**两种检测方法:**

| 方法 | 阈值 | 说明 |
|------|------|------|
| **FACK (包号距离)** | `largest_acked - packno > 3` | 如果后面 3 个包都被确认,则认为丢失 |
| **时间阈值** | `now - sent_time > 1.25 × SRTT` | 超过 RTT 的 125% 仍未确认 |

#### 超时计算
```c
delay = ctl->sc_loss_to;  // 🔥 使用检测到的丢包超时时间
```

#### 示例
```
发送: #1, #2, #3, #4, #5
收到: ACK{#3, #4, #5}  ← #1, #2 未确认

检测: #1, #2 被标记为丢失 (FACK 检测)
      因为 largest_acked(5) - packno(1) = 4 > 3

重传: 立即重传 #1, #2
```

---

### 3️⃣ **RETX_MODE_TLP** (Tail Loss Probe - 尾部丢包探测)

#### 含义
**主动探测** 机制,在怀疑最后几个包丢失时发送探测包。

#### 触发条件
```c
if (ctl->sc_n_tlp < 2)
    return RETX_MODE_TLP;  // 🔥 TLP 次数 < 2
```

**条件:**
- 没有检测到明确的丢包 (`sc_loss_to == 0`)
- TLP 探测次数少于 2 次

#### 超时计算
```c
static lsquic_time_t calculate_tlp_delay (lsquic_send_ctl_t *ctl) {
    srtt = lsquic_rtt_stats_get_srtt(&ctl->sc_conn_pub->rtt_stats);
    
    if (ctl->sc_n_in_flight_all > 1)
        delay = 10000;  // 🔥 最小 10ms (多个包在途)
    else
        delay = srtt + srtt / 2 + max_peer_ack_delay;  // 🔥 1.5×SRTT + 对端延迟
    
    if (delay < 2 * srtt)
        delay = 2 * srtt;  // 🔥 至少 2×SRTT
    
    return delay;
}
```

**TLP 的作用:**
- **避免长时间等待**: 不等 RTO,主动探测
- **快速恢复**: 触发对端发送 ACK
- **限制次数**: 最多 2 次,防止过度发送

#### 示例
```
T=0ms:    发送 #1, #2, #3
T=50ms:   收到 ACK{#1, #2}  ← #3 可能丢失
T=100ms:  TLP 超时 → 发送探测包 #4 (重传 #3 的内容)
T=110ms:  收到 ACK{#3} → #3 没丢,只是延迟了
```

---

### 4️⃣ **RETX_MODE_RTO** (Retransmission Timeout - 重传超时)

#### 含义
**最后的防线**,当其他机制都失败时使用,触发最激进的恢复。

#### 触发条件
```c
default:
    assert(rm == RETX_MODE_RTO);  // 🔥 fallback 模式
    return RETX_MODE_RTO;
```

**条件:**
- TLP 次数已达上限 (≥ 2)
- 没有检测到明确的丢包

#### 超时计算
```c
static lsquic_time_t calculate_packet_rto (lsquic_send_ctl_t *ctl) {
    delay = get_retx_delay(&ctl->sc_conn_pub->rtt_stats);
    
    unsigned exp = ctl->sc_n_consec_rtos;
    if (exp > MAX_RTO_BACKOFFS)
        exp = MAX_RTO_BACKOFFS;  // 🔥 最多 10 次退避
    
    delay = delay * (1 << exp);  // 🔥 指数退避: 2^n
    
    if (delay > MAX_RTO_DELAY)
        delay = MAX_RTO_DELAY;   // 🔥 上限 60s
    
    return delay;
}

// 基础延迟计算
static lsquic_time_t get_retx_delay (const struct lsquic_rtt_stats *rtt_stats) {
    srtt = lsquic_rtt_stats_get_srtt(rtt_stats);
    
    if (srtt) {
        delay = srtt + 4 * lsquic_rtt_stats_get_rttvar(rtt_stats);  // 🔥 SRTT + 4×RTTVAR
        if (delay < MIN_RTO_DELAY)
            delay = MIN_RTO_DELAY;  // 最小 200ms
    } else {
        delay = DEFAULT_RETX_DELAY;  // 默认 500ms
    }
    
    return delay;
}
```

**RTO 的影响:**
- **减小拥塞窗口**: `cci_timeout()` 触发
- **指数退避**: 每次 RTO 时间翻倍
- **连接可能超时**: 多次 RTO 后可能关闭连接

#### 示例
```
T=0ms:     发送 #1, #2, #3
T=100ms:   TLP #1 → 发送探测包 #4
T=200ms:   TLP #2 → 发送探测包 #5
T=700ms:   RTO (500ms) → 重传所有未确认的包
           cwnd /= 2 (拥塞窗口减半)
T=1700ms:  RTO (1000ms, 指数退避) → 再次重传
T=3700ms:  RTO (2000ms) → ...
```

---

## 🔄 模式选择流程图

```
开始
  ↓
握手完成了吗? ───No──→ 有未确认握手包? ───Yes──→ [RETX_MODE_HANDSHAKE]
  ↓ Yes                      ↓ No
检测到丢包了吗? ───Yes──→ [RETX_MODE_LOSS]
  ↓ No
TLP 次数 < 2? ───Yes──→ [RETX_MODE_TLP]
  ↓ No
[RETX_MODE_RTO] (最后防线)
```

## 📊 四种模式对比

| 模式 | 优先级 | 超时计算 | 退避策略 | 拥塞控制 | 适用场景 |
|------|--------|---------|---------|---------|---------|
| **HANDSHAKE** | 🔥🔥🔥🔥 | `max(1.5×SRTT, 10ms) × 2^n` | 指数 | ❌ 不触发 | 连接建立 |
| **LOSS** | 🔥🔥🔥 | `sc_loss_to` (检测值) | 无 | ✅ 触发 | 检测到丢包 |
| **TLP** | 🔥🔥 | `max(2×SRTT, 10ms)` | 限 2 次 | ❌ 不触发 | 尾部探测 |
| **RTO** | 🔥 | `(SRTT + 4×RTTVAR) × 2^n` | 指数 | ✅ 触发 | 最后防线 |

## 🎯 UTP 实现建议

```cpp
namespace eular {
namespace utp {

enum class RetxMode : uint8_t {
    Handshake,  // 握手快速重传
    Loss,       // 基于丢包检测
    TLP,        // 尾部丢包探测
    RTO         // 超时重传
};

class SendControl {
private:
    RetxMode getRetxMode() const {
        // 1. 握手优先
        if (!connPub_->handshakeDone && hasUnackedHandshakePackets())
            return RetxMode::Handshake;
        
        // 2. 明确丢包
        if (lossTo_ > 0)
            return RetxMode::Loss;
        
        // 3. TLP 探测
        if (nTlp_ < 2)
            return RetxMode::TLP;
        
        // 4. RTO 兜底
        return RetxMode::RTO;
    }
    
    uint64_t calculateRetxDelay(RetxMode mode) const {
        switch (mode) {
        case RetxMode::Handshake:
            return calculateHandshakeDelay();
        case RetxMode::Loss:
            return lossTo_;
        case RetxMode::TLP:
            return calculateTlpDelay();
        case RetxMode::RTO:
            return calculateRtoDelay();
        }
    }
};

} // namespace utp
} // namespace eular
```
