# UTP 协议设计文档（实现对齐版）

> 更新时间：2026-03-24  
> 目标：对齐当前仓库实现，形成“可实现、可演进”的轻量 UDP 传输协议规范。  
> 定位：类似 QUIC 的设计思路，但实现更轻量，优先满足低复杂度和工程可迭代性。

> OpenSpec 工作流说明：当前实现基线已经按 `connection-lifecycle`、`stream-transport`、`ack-and-loss-recovery`、`path-validation`、`session-resumption-zero-rtt` 五个 capability 落入 OpenSpec。后续变更若涉及这些行为，应修改对应 capability spec 或新增 capability change，而不是仅更新本说明文档。

---

## 1. 设计目标

### 1.1 目标
- 基于 UDP 提供可靠传输、拥塞控制、流复用的基础能力。
- 使用 Connection ID（CID）弱化五元组变化带来的连接中断。
- 在当前版本预埋 NAT Rebinding / 连接迁移能力（路径验证 + 会话票据）。
- 保持实现简单，不一次性引入完整 QUIC 全量机制。

### 1.2 非目标（当前版本）
- 不实现完整 ICE / STUN / TURN 穿透流程。
- 不实现复杂多路径调度与路径优先级竞争。
- 不实现 QUIC 级别所有帧和所有恢复细节。

## 5. 核心帧格式（当前实现重点）

### 5.1 Stream

```c++
struct {
    uint8_t     frame_type;
    uint8_t     stream_flag;
    uint16_t    stream_data_length;
    uint32_t    stream_id;
    uint64_t    stream_offset;
    uint8_t[]   stream_data;
}
```

要点：
- `stream_id` 连接内唯一，不复用。
- `stream_offset` 用于乱序重组。
- `stream_flag` 的 `FIN` 表示流数据发送结束。

### 5.1.1 Stream 级优先级（待评审）

现状：
- 当前实现已具备多流收发，但发送侧尚无显式 stream 级优先级调度器。
- 在大流持续占用发送窗口时，交互型小流可能出现尾延迟抬升。

目标：
- 为每个 stream 提供可配置优先级（建议 0~7，数值越小优先级越高）。
- 在不破坏现有 ACK/重传/拥塞控制闭环的前提下，优先保障高优先级流时延。

设计评审范围（本阶段不编码）：
1. 调度模型：严格优先级（Strict Priority）或加权公平（WFQ/DRR）。
2. 饥饿保护：低优先级流的最小服务保证与突发上限。
3. 交互边界：与 `cwnd`、pacer、重传队列的仲裁顺序。
4. 可观测性：至少输出模式切换、选流、回压与调度累计计数，支持线上排障。

已落地的对外可观测字段（`Connection::Statistic`）：
1. `scheduler_select_total`：总选流次数。
2. `scheduler_select_disabled`：DISABLED 模式选流次数。
3. `scheduler_select_strict`：STRICT 模式选流次数。
4. `scheduler_select_drr`：DRR 模式选流次数。
5. `scheduler_strict_aging_promoted`：STRICT 中 aging 提升次数。
6. `scheduler_would_block`：选中流发送返回 `WOULD_BLOCK` 次数。
7. `scheduler_empty_rounds`：flush 未选中可发 stream 的轮次。
8. `scheduler_mode_switches`：调度模式热切换次数。
9. `scheduler_drr_refills`：DRR deficit 补充次数。
10. `scheduler_drr_consumes`：DRR deficit 消耗次数。

建议分阶段：
1. 文档阶段（已完成）：固化需求、目标、评审议题。
2. 评审阶段：形成最终调度方案与兼容策略（默认关闭或默认中性优先级）。
3. 实现阶段：落地调度器与配置项，补齐单测/集成回归。

### 5.5 PathChallenge（NAT 预留关键）

```c++
struct {
    uint8_t     frame_type;
    uint8_t     data[8];
}
```

规则：
- `data` 必须不可预测随机值；
- 用于验证“对端是否真实可达该新路径”。

---

### 5.6 PathResponse（NAT 预留关键）

```c++
struct {
    uint8_t     frame_type;
    uint8_t     data[8]; // 原样回显 challenge data
}
```

规则：
- 必须回显最近收到的 `PathChallenge.data`；
- 不匹配应丢弃，不改变路径状态。

### 5.7 MTU 探测策略（DPLPMTUD）

目标：在不依赖 ICMP 的前提下，以更少 RTT 收敛到可用 PMTU，并在路径突变时避免黑洞持续。

规则：
1. 首阶段采用固定梯队探测：`1380 -> 1450 -> 1492 -> 1500`。
2. 梯队中不包含 `1280`：最小安全包长由握手/首包路径验证承担；若最小安全包长不可达，连接应走异常收敛。
3. 当梯队出现“低档成功、高档失败”时，进入二分区间继续探测。
    - 示例：`1450` 成功、`1492` 失败，则二分区间为 `[1451, 1491]`。
4. 配置项 `mtu_probe_step` 语义为“二分终止精度阈值”，不再作为线性上探步长。
    - 当 `high - low <= mtu_probe_step` 时，二分收敛结束。
5. 触发 MTU 黑洞判定后，需立即回退到最小安全 MTU `1280`（IPv4/IPv6 统一），并在冷静期后重新执行梯队探测。

可观测性要求：
1. 记录探测阶段（梯队/二分/稳定/黑洞恢复）迁移日志。
2. 记录黑洞触发次数与最近一次回退目标 MTU。

### 5.7 SessionToken（NAT 预留关键）

```c++
struct {
    uint8_t     frame_type;
    uint8_t     token_size;
    uint16_t    token_validity_period; // seconds
    uint8_t[]   token;
}
```

用途：
- 握手阶段给对端发放短期票据；
- 后续网络重绑定时作为路径校验辅助凭证；
- 减少伪造源地址导致的资源消耗。

0-RTT 约束：
- 当 `UTP_TYPE_0RTT` 承载会话恢复时，`SessionToken` 必须位于 payload 首帧，且保持可直接解析。
- 若本次 0-RTT 使用早期数据加密，则仅 `SessionToken` 帧保持明文；后续应用数据帧（如 `Stream`）进入早期 AEAD 保护区。
- 接收端必须先解析 `SessionToken`，恢复票据中的加密模式与恢复密钥材料，再解密后续早期数据帧。

SessionResumptionState（对外封装约束）：
- 对应用仅暴露 opaque 字符串，不暴露解析结构。
- 编码使用标准 Base64（非 URL-safe 变体）。
- 封装内部至少包含：`version`、`encryption_mode`、`session_ticket`、`resumption_psk`、`expires_at`。
- 封装必须经过 AEAD 保护，防止本地串内容被篡改后静默生效。
- SDK 不负责持久化，应用负责存储与恢复该字符串。
- 直接票据式 `connect0Rtt` 仅适用于非加密场景；加密恢复统一使用 `connect0RttWithState`。
- 应提供直接导出 `SessionToken` 的接口，供非加密 `connect0Rtt` 使用。

## 6. 连接与路径状态机（轻量版）

### 6.1 0-RTT 首发规则

`UTP_TYPE_0RTT` 首包采用“首帧明文恢复信息 + 后续帧加密业务数据”的布局：

1. `SessionToken` 必须作为 payload 第一帧出现，用于接收端无歧义定位边界并优先完成验票。
2. `SessionToken` 之后的早期应用帧可使用 0-RTT 恢复得到的 AEAD 上下文加密。
3. 若票据要求加密但接收端无法恢复对应早期密钥，则必须拒绝 0-RTT，而不是退化为明文解析后续应用帧。
4. 后续实现若演进为“双首发包”模型，应显式替换本规则；在此之前默认遵循本节约束。
5. `connect0RttWithState` 的返回值仅表达前置校验失败（如 Base64 解码失败、AEAD 解密失败、本地过期等）；服务端拒绝由异步回调报告。
6. 应提供“SessionTokenReady”通知机制，应用收到通知后可主动调用 `exportSessionToken` 或 `exportSessionResumptionState` 导出最新材料。
7. 普通 `ConnectInfo` 不承载 `session_ticket`、`resumption_psk` 等 0-RTT 恢复材料，避免与常规建连语义混杂。

### 6.1 连接状态（Connection）
- `WaitSendInitial`
- `InitialSent`
- `HandshakeSent`
- `HandshakeReceived`
- `Connected`
- `CloseSent`
- `CloseReceived`
- `PtoTimedWait`
- `Disconnected`

对应代码：`src/context/connection_impl.h`。

### 6.2 路径状态（Path）
- `Unknown`：无可用路径信息。
- `Validated`：当前路径已验证可达。
- `Validating`：检测到地址变化，等待验证。
- `Failed`：验证失败（超时且达到重试上限）。

对应代码：`src/util/network_path.h`。

## 7. NAT Rebinding 处理流程（当前版本）

### 7.1 触发条件
- 收包时发现远端地址（IP/端口）变化；
- CID 仍匹配当前连接。

### 7.2 执行流程
1. 路径状态切换到 `Validating`；
2. 发送 `PathChallenge`（8B 随机 data）；
3. 启动验证计时器；
4. 收到 `PathResponse` 且 data 匹配：
   - 路径切换到 `Validated`；
   - 清理 challenge 重试状态；
5. 超时未匹配：
   - 未超重试上限：继续 challenge；
   - 达到上限：切换 `Failed`。

### 7.3 默认参数
- `challenge_timeout_ms = 1500`
- `max_challenge_retries = 3`

可在后续版本下放到 `Config`。

### 7.4 实现现状校对记录（2026-03）

以下条目用于“协议校对阶段”的现状确认，结论来自当前代码实现，用于后续评审通过后再做协议增强开发。

1. 当前已存在完整路径校验机制，不是占位逻辑。
2. 校验触发点：连接已建立后，若检测到对端地址（IP/端口）变化，会进入 `Validating`。
3. 校验报文：使用 `PathChallenge(data[8])` 与 `PathResponse(data[8])`，`PathResponse` 必须原样回显 challenge 数据。
4. 校验通过条件：收到匹配的 `PathResponse` 后，路径状态转为 `Validated`，并清理重试与定时器状态。
5. 超时与重试：挑战超时后若未到重试上限会重发 challenge；达到上限后路径状态转为 `Failed`。
6. 失败后行为：保守迁移模式下，`candidate_path` 校验失败会回退/保持 `active_path`，连接不因候选路径失败而关闭。
7. 放大限制：在 `Validating` 阶段发送受限，采用 `out_bytes <= in_bytes * 3 + credit` 约束；仅少量控制帧可无条件发出（如 PathChallenge/PathResponse/Ping/Ack/HandshakeDone/ConnectionClose）。

备注：
- 上述机制满足“地址变化后先验路再放量发送”的最小安全闭环。
- 当前实现尚未引入独立 `PathToken` 帧；若协议评审决定补充 token 化路径授权，应作为后续增量特性实现，不影响本节现状结论。

### 7.5 落地实施方案（已执行状态）

本节用于记录“已落地行为 + 演进边界”，与当前代码行为保持一致。

#### 阶段 A：SessionToken 优先（优先级 P0）

目标：仅围绕 0-RTT 重连打通票据闭环，先不引入 PathToken 扩展。

方案：
1. 服务端在连接进入 `Connected` 后下发 `SessionToken`，用于后续 `connect0Rtt`。
2. 票据用途限定为“下一次连接的 0-RTT 重连材料”，不承担路径迁移授权语义。
3. 仅保留 SessionToken 相关配置与统计项；本阶段不新增 PathToken 相关变量。
4. 客户端对收到的 SessionToken 仅做基础防护（地址有效、长度正确、有效期窗口）后入缓存，不在客户端执行 `open` 解密验票。
5. 票据真实性、地址绑定与时效性的强校验由服务端在 0-RTT 接入点统一执行。
6. `SessionTokenReady`、`exportSessionToken`、`exportSessionResumptionState` 由 Connection 级接口承载，不再由 Context 统一缓存导出。

验收标准：
1. 客户端可稳定获取并复用 `SessionToken` 发起 0-RTT 重连。
2. 客户端不因缺少服务端密钥而拒收 SessionToken（按 opaque 数据缓存）。
3. 服务端对无效票据（篡改/过期/地址不匹配）拒绝 0-RTT，并可回退常规连接流程。
4. 本阶段无 PathToken 依赖，无 PathToken 开关。

#### 阶段 B：双路径状态模型（已落地，默认保守模式）

目标：在不破坏业务连续性的前提下完成路径迁移闭环，实现“验证成功后再切换”。

模型定义：
1. `active_path`：当前承载业务数据的已验证路径。
2. `candidate_path`：新地址对应的待验证路径，仅用于 `PathChallenge/PathResponse` 探测。

迁移规则：
1. 检测到地址变化后，不立即切换业务发送路径。
2. 探测期间业务数据继续走 `active_path`，探测帧走 `candidate_path`。
3. `candidate_path` 验证成功后，再原子切换为新的 `active_path`。
4. `candidate_path` 验证失败时，维持原 `active_path`，不影响业务连续性。

说明：
1. 当前已按该模型完成核心实现：探测帧发往 `candidate_path`，业务数据维持 `active_path`，校验通过后再切换。
2. 由于当前决定不引入 PathToken，本模型仍以 `PathChallenge(8B)`/`PathResponse` 为核心验证手段。
3. `Config.path_migration_mode` 已预留配置入口：默认 `kPathMigrationConservative`。
4. `kPathMigrationAggressive` 仅保留开关位与配置入口，当前未启用。
5. TODO：激进策略完整实现前，代码路径必须强制按保守策略执行，避免配置误开导致行为不一致。

#### 阶段 C：发布策略（优先级 P0）

1. 当前迭代灰度范围：`session_token_auto_issue` + 保守路径迁移实现。
2. 监控维度分两组：
    - 0-RTT：接受率、拒绝原因、重连成功率。
    - 路径迁移：candidate 验证成功率、验证超时率、迁移后首 RTT 波动。
3. 激进迁移模式不开启，不进入灰度矩阵。

#### 阶段 D：API 断代与研发节奏（评审已确认）

1. 接受一次性断代：UTP 初版不保留 Context 级 SessionToken 兼容补丁，统一切换到 Connection 级接口。
2. 本阶段“先协议后测试”：在协议细节冻结前，不扩展测试面，避免因协议反复修改产生高额返工。
3. 协议冻结后集中补齐测试：按冻结版接口一次性补全单测与集成测试，并据此进入灰度。

执行顺序：
1. 优先完成协议评审与冻结（SessionToken/0-RTT 语义、双路径模型边界）。
2. 协议冻结后再进行测试补齐与示例收敛。
3. 测试通过后进入小流量灰度。

#### 本轮落地结论

1. 当前迭代已落地 SessionToken 用于 0-RTT 重连（Connection 级导出与通知）。
2. 双路径状态模型已落地保守模式：验证成功后切换，失败回退/保持 active。
3. 激进模式仅保留配置入口与 TODO，不启用。
4. 当前实施顺序更新为 A -> B -> C -> D。

    - 过期票据拒绝；
    - 合法票据重连成功与回退路径正确。
3. 增加可观测性字段：
    - path_validation_started/succeeded/failed；
    - 0rtt_accept/0rtt_reject_reason。

P1（小流量灰度阶段）：

    - 0-RTT 接受率与错误码分布；

    - 0-RTT 拒绝率异常升高；

## 8. 安全与放大限制（建议约束）

为降低源地址伪造放大风险，建议在“路径未验证”阶段启用：

- `bytes_out <= 3 * bytes_in`
- 未通过路径验证前，不发送大体积应用数据
- 只允许发送最小必要控制帧（Ack/PathResponse/受限重传）

此策略与 NAT 版本保持一致，可直接复用。

## 10. 为 NAT 穿透版本预留的扩展点

下一版本（NAT 穿透版）建议在本版本基础上增加：

1. **候选地址发现**：STUN 获取 server-reflexive 地址；
2. **候选交换**：通过已有可靠流交换 candidate 集；
3. **并行打洞**：对候选地址并发发送探测包（含 token + challenge）；
4. **路径择优**：以 RTT、丢包率、可达性稳定度选择主路径；
5. **回退策略**：直连失败回落 relay（如 TURN/自建中继）；
6. **安全增强**：票据绑定 CID/地址族/时间窗，防重放。

## 11. 兼容性约束

- 新增帧类型必须保持向后兼容：未知帧可忽略或触发协商失败。
- 关键状态迁移需可观测（日志 + 统计计数）。
- 帧编解码应遵循“先校验长度、再反序列化、最后更新状态”的顺序。
