# libutp 默认值与调参指南

> 更新时间：2026-05-09
> 目标：给出握手、ACK、自适应策略、Stream 缓冲与调度参数的当前默认值与推荐调优方式，便于线上按场景调优。

## 1. 配置项总览

| 配置项 | 默认值 | 作用 | 过小风险 | 过大风险 |
|---|---:|---|---|---|
| `handshake_timeout` | 800 ms | 握手首轮超时基准，同时通过 TP 向对端声明建议值 | 弱网或高 RTT 下首轮等待过短，易触发不必要重试 | 尾部收敛和握手失败判定变慢，占用资源时间变长 |
| `handshake_max_retries` | 2 | 握手超时后的最大重试次数（本地参数，不进入 TP） | 极端丢包下建连成功率下降 | 真实失败连接收敛变慢 |
| `ack_every_n_packets` | 4 | 每累计 N 个 ack-eliciting 包触发 ACK | ACK 频繁，控制面开销升高 | ACK 稀疏，丢包恢复变慢 |
| `ack_delay` | 50 ms | 接收端 ACK 延迟上限 | 高频 ACK，CPU/带宽开销上升 | 重传检测与恢复延迟增加 |
| `mtu_probe_step` | 16 bytes | MTU 二分探测收敛精度阈值（非线性步长） | 收敛精度过细，探测轮次增加 | 收敛过粗，最终 MTU 可能偏保守 |
| `stream_send_buffer_limit` | 256 KB | 单个 Stream 本地发送缓存上限 | 容易触发应用写阻塞 | 单流突发占用内存大 |
| `stream_unacked_data_limit` | 256 KB | 连接内 Stream 在途未确认数据总门限 | 吞吐受限，写阻塞增加 | 未确认发送堆积，内存与时延抖动增大 |
| `stream_enable_coalescing` | true | tiny write 聚合开关 | 关闭后小写入更易形成小包 | 打开时会引入极短发送等待 |
| `stream_min_payload_before_immediate_send` | 1200 bytes | 触发“立即发送”的最小 payload 阈值 | 过小会降低聚合收益 | 过大可能增加小流发送等待 |
| `stream_coalesce_delay_us` | 1000 us | tiny write 聚合窗口 | 过小聚合收益有限 | 过大增加尾时延 |
| `bbr_init_cwnd_mss` | 32 | BBR 初始拥塞窗口（MSS） | 冷启动吞吐偏低 | 初始突发可能增加排队与丢包 |
| `bbr_min_cwnd_mss` | 4 | BBR 最小拥塞窗口（MSS） | 丢包后恢复更慢 | 丢包期窗口下限偏高 |
| `bbr_startup_high_gain` | 2.885 | BBR STARTUP 增益 | 过小导致带宽爬升慢 | 过大导致探测过激进 |
| `bbr_cwnd_gain` | 2.0 | BBR PROBE_BW cwnd 增益 | 过小限制吞吐上限 | 过大增加队列堆积概率 |
| `bbr_startup_growth_target` | 1.25 | BBR 判定“带宽有增长”的倍率阈值 | 过低易过早退出 STARTUP | 过高可能长时间停留 STARTUP |
| `bbr_startup_full_bw_rounds` | 3 | BBR 连续无增长后退出 STARTUP 的轮数 | 过小易保守 | 过大可能过冲 |
| `bbr_probe_rtt_ms` | 200 ms | BBR PROBE_RTT 最短驻留时长 | 过小 min_rtt 采样不稳定 | 过大影响吞吐 |
| `bbr_min_rtt_expiry_ms` | 10000 ms | BBR min_rtt 过期时间 | 过小频繁进 ProbeRTT | 过大 min_rtt 可能陈旧 |
| `cubic_beta` | 0.7 | CUBIC 丢包回退系数 | 过小导致过度降窗、吞吐下降 | 过大导致回退不充分、丢包恢复变慢 |
| `cubic_c` | 0.4 | CUBIC 曲线常数 | 过小导致窗口爬升慢 | 过大导致窗口爬升激进 |
| `cubic_init_cwnd_mss` | 32 | CUBIC 初始拥塞窗口（MSS） | 冷启动吞吐偏低 | 初始突发可能增加排队/丢包 |
| `cubic_min_cwnd_mss` | 4 | CUBIC 最小拥塞窗口（MSS） | 回退后恢复更慢 | 丢包期窗口下限偏高 |
| `connection_scheduler_mode` | `WDRR` | 连接级调度模式（Disabled/Strict/WDRR） | Disabled/Strict 在高并发下可能出现连接饥饿 | WDRR 配置不当可能导致短时抖动 |
| `connection_wdrr_quantum` | 1200 bytes | 每轮为连接补充的 deficit | 过小会降低大流吞吐 | 过大可能削弱连接间公平性 |
| `connection_wdrr_deficit_cap` | 512 KB | 单连接 deficit 上限 | 过小会让突发连接频繁回补 | 过大可能放大短时不公平 |

## 2. 握手超时与重试

当前实现语义：

1. `handshake_timeout` 是本端握手首轮超时基准。
2. `handshake_timeout` 会通过 `TransportParams` 发给对端，作为建议值。
3. 收到对端 `handshake_timeout` 后，握手相关定时采用 `min(local, peer)`。
4. `handshake_max_retries` 只在本地生效，不进入 TP。
5. 第 `k` 轮等待时间按指数退避计算：

`round_timeout_k = effective_handshake_timeout * 2^k`

其中：

- `effective_handshake_timeout = handshake_timeout`（收到对端 TP 前）
- `effective_handshake_timeout = min(local_handshake_timeout, peer_handshake_timeout)`（收到对端 TP 后）

按当前默认值：

- 第 0 轮：`800 ms`
- 第 1 轮：`1600 ms`
- 第 2 轮：`3200 ms`

因此默认总预算大约为 `5600 ms`。

推荐调参：

1. 局域网或机房内低 RTT：
- `handshake_timeout = 300 ~ 800 ms`
- `handshake_max_retries = 1 ~ 2`

2. 常规公网：
- `handshake_timeout = 800 ~ 1500 ms`
- `handshake_max_retries = 2`

3. 高 RTT、突发丢包、跨境或移动网络：
- `handshake_timeout = 1500 ~ 3000 ms`
- `handshake_max_retries = 2 ~ 3`

调参原则：

- 优先先调 `handshake_timeout`，再看是否需要增加 `handshake_max_retries`。
- 若问题是“首轮太快超时”，优先增大 `handshake_timeout`。
- 若问题是“偶发突发丢包导致最后几轮才成功”，再增加 `handshake_max_retries`。

## 3. ACK 与自适应策略

当前实现要点：

1. 握手阶段会携带 `AckFrequency`。
2. 建连后按 `srtt` 自适应 `max_ack_delay_ms`。
3. 更新限频：最小更新间隔 3000ms，变化不足阈值不触发更新。

推荐公式：

`max_ack_delay_ms = clamp(srtt / 4, 10ms, 100ms)`（并按 5ms 量化）

## 4. 典型场景推荐值

### 3.w BBR 专项调参（新增）

生效前提：`cc_algorithm = 0` 或 `cc_algorithm = 1`。

1. 稳定公网默认建议：
- `bbr_init_cwnd_mss = 32`
- `bbr_min_cwnd_mss = 4`
- `bbr_startup_high_gain = 2.885`
- `bbr_cwnd_gain = 2.0`
- `bbr_startup_growth_target = 1.25`
- `bbr_startup_full_bw_rounds = 3`
- `bbr_probe_rtt_ms = 200`
- `bbr_min_rtt_expiry_ms = 10000`

2. 高带宽高 RTT（BDP 大）链路：
- 可适当提高 `bbr_init_cwnd_mss` 到 `40 ~ 64`，加快冷启动。
- 若启动过冲明显，回调 `bbr_startup_high_gain` 到 `2.2 ~ 2.6`。

3. 高抖动或易拥塞链路：
- 可降低 `bbr_startup_high_gain` 到 `2.0 ~ 2.5`。
- 可提高 `bbr_startup_growth_target` 到 `1.3 ~ 1.4`，更谨慎退出 STARTUP。
- 可增加 `bbr_startup_full_bw_rounds` 到 `4 ~ 6`，降低误判风险。

### 3.x CUBIC 专项调参（新增）

生效前提：`cc_algorithm = 2`。

1. 稳定公网默认建议：
- `cubic_beta = 0.7`
- `cubic_c = 0.4`
- `cubic_init_cwnd_mss = 32`
- `cubic_min_cwnd_mss = 4`

2. 高带宽高 RTT（BDP 大）链路：
- 可将 `cubic_c` 提高到 `0.5 ~ 0.8`，提升拥塞避免阶段爬升速度。
- 观察重传率与尾时延，若抖动明显再回调。

3. 易拥塞或抖动链路：
- 可将 `cubic_beta` 下调到 `0.6 ~ 0.68`，增强丢包后的降窗幅度。
- 可适当降低 `cubic_init_cwnd_mss` 到 `16 ~ 24`，减小冷启动突发。

4. 约束关系：
- `cubic_beta` 有效范围 `(0, 1)`，超出范围会回退默认值。
- `cubic_c` 有效范围 `(0, 2]`，超出范围会回退默认值。
- `cubic_init_cwnd_mss` 会被自动钳制为 `>= cubic_min_cwnd_mss`。

### 3.0 MTU 探测说明（新增）

1. 当前 MTU 探测采用“梯队 + 二分”策略，梯队依次为 `1380, 1450, 1492, 1500`。
2. `mtu_probe_step` 用于控制二分收敛停止条件（`high-low <= step`），不是每轮固定累加值。
3. 若路径突变导致黑洞，系统会回退到安全 MTU 后再重启探测；此时可适当调大 `mtu_probe_interval` 降低抖动。

### 3.1 低时延交互（游戏/IM）

- `handshake_timeout = 300 ~ 800 ms`
- `handshake_max_retries = 1 ~ 2`
- `ack_every_n_packets = 2 ~ 4`
- `ack_delay = 10 ~ 25 ms`
- `stream_send_buffer_limit = 128 ~ 256 KB`
- `stream_unacked_data_limit = 128 ~ 256 KB`

目标：优先降低恢复时延与交互抖动。

### 3.2 高吞吐传输（文件/媒体分发）

- `handshake_timeout = 800 ~ 1500 ms`
- `handshake_max_retries = 2`
- `ack_every_n_packets = 8 ~ 16`
- `ack_delay = 25 ~ 60 ms`
- `stream_send_buffer_limit = 256 ~ 1024 KB`
- `stream_unacked_data_limit = 512 ~ 2048 KB`

目标：提升吞吐并降低控制面比例。

### 3.3 弱网高 RTT（跨境/移动网络）

- `handshake_timeout = 1500 ~ 3000 ms`
- `handshake_max_retries = 2 ~ 3`
- `ack_every_n_packets = 4 ~ 8`
- `ack_delay = 40 ~ 100 ms`
- `stream_send_buffer_limit = 256 ~ 512 KB`
- `stream_unacked_data_limit = 256 ~ 1024 KB`

目标：兼顾恢复灵敏度与 ACK 开销，避免过度重传。

## 5. 调参步骤建议

1. 先确认握手问题还是数据面问题；不要一上来同时调握手和 ACK。
2. 若建连成功率不足，先调 `handshake_timeout`，再调 `handshake_max_retries`。
3. 再固定 `stream_send_buffer_limit`，观察写阻塞率与内存曲线。
4. 再调 `stream_unacked_data_limit`，观察 unacked backlog 和 RTT 抖动。
5. 最后微调 ACK 参数（`ack_every_n_packets` / `ack_delay`）。
6. 每次只改一组参数，并保留至少 24 小时稳定性观察窗口。

## 6. 验证指标建议

1. 建连成功率与 `connect timeout` 比例。
2. 握手完成耗时分位数（P50/P95/P99）。
3. 写阻塞率（`UTP_ERR_WOULD_BLOCK` 占比）。
4. 发送侧 backlog（`bytesOutTotal`）峰值与分位数。
5. RTT / RTTVar / 重传率。
6. 进程 RSS 与单连接内存峰值。
7. 吞吐与尾时延（P95/P99）。

## 7. 注意事项

1. `handshake_timeout` 不是“整次建连总预算”，而是首轮超时基准；总预算受指数退避和 `handshake_max_retries` 共同影响。
2. `handshake_max_retries` 是纯本地策略参数，不通过 TP 协商。
3. `stream_unacked_data_limit` 是连接内 Stream 在途数据门限，不是单流上限。
4. 多流场景下若某流持续大包写入，其他流可能短时受限；应结合业务做流优先级策略。
5. 如需在线热更新，建议先灰度小流量并观察 15~30 分钟指标后再全量。
