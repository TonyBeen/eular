## 3. 已实现功能

### 3.1 连接与事件模型

当前仓库已经落实以下连接能力：

- Context 提供 bind、connect、accept 等入口。
- 主动建连成功后可进入 Connected。
- 被动建连采用两阶段接入：先收到请求，再由应用显式 accept 放行。
- OnConnected、OnConnectError、OnConnectionClosed 等回调已具备可用语义。
- OnNewConnection 可用于被动连接的准入决策。

当前实现特点：

- 被动连接不是“收到 Initial 就直接变成正式连接”，而是 pending -> accept -> 等待 HandshakeDone -> 晋升 Connected。
- 同步前置失败与异步握手失败都统一汇总到连接结果事件中。
- HandshakeDone 不再按“发送成功即收敛”处理，而是以 ACK 覆盖为准；超时会继续重发。
- pending 阶段对未携带 HandshakeDone 的后续包采用小窗口缓存，收到 HandshakeDone 后回放，降低尾部丢包连锁影响。

### 3.2 Stream 数据面

当前已落实：

- createStream / getStream
- 流收发缓冲
- read / write
- FIN / RESET_STREAM 基本语义
- 零拷贝 acquire / commit 接口
- 流 ID 准入与活跃流配额校验

说明：

- 当前流模型已经不是“只有帧编解码”，而是完整接入了连接主流程。
- 流关闭、可读、异常终止在 API 层已有对外语义。

### 3.3 Stream 优先级与调度

当前已落实：

- Stream::setPriority() / priority() 接口
- 默认优先级与优先级范围定义
- Config.stream_scheduler_mode
- Config.stream_aging_threshold
- Config.stream_aging_step
- Config.stream_drr_quantum
- Config.stream_drr_deficit_cap
- STRICT 与 DRR 两种调度模式
- 连接级调度统计字段

当前结论：

- stream 优先级能力已经落地，不再属于“只有设计没有实现”的阶段。
- 剩余工作主要是高负载收益验证、指标看板化和参数进一步调优。

### 3.4 包解析与发送对象生命周期

当前已落实：

- 固定头解析
- payload 长度校验
- 帧遍历
- PacketOut 池化管理
- PacketIn 池化接入主收包路径
- PacketOut attempt 链记录每次发送尝试

实现意义：

- 发送侧已经能保留“一个逻辑包多次发送尝试”的轻量历史。
- 接收与发送对象都已有可复用的内存生命周期管理。

### 3.5 ACK、丢包检测与重传

当前已落实：

- Ack 编解码
- ACK 范围解析
- ACK 驱动的 inflight/unacked 回收
- ACK timer
- 包阈值 ACK
- 乱序快速 ACK
- ACK piggyback
- FACK + 重排序阈值丢包检测
- 丢包队列管理
- 使用新包号的重传策略

当前结论：

- 发送恢复闭环已经形成，不再是单向 ACK 解码。
- AckFrequency 不只是帧定义，已经参与 ACK 策略与发送侧重排序阈值联动。

### 3.5.1 握手参数职责收敛

当前已落实：

- TransportParams 已从 Crypto 中拆出为独立帧
- 握手阶段 Initial 和 Handshake 都发送 TransportParams 与 AckFrequency
- `TransportParams.max_ack_delay` 已删除，ACK 最大延迟只保留在 AckFrequency
- 运行期发送侧超时计算改为采用对端 AckFrequency 宣告的 `max_ack_delay_ms`
- 主动侧 HandshakeDone 定时优先采用对端 TransportParams 中的 `handshake_timeout / 3`

当前语义：

- Config 中的握手和 ACK 参数仅作为本端握手时发给对端的声明值。
- 连接建立后，本端行为采用对端发来的 TransportParams 与 AckFrequency。

### 3.5.2 协议 v2 握手辅助与确认语义（进行中）

已确认的设计目标：

- AckFrequency 删除 `timestamp` 字段，保留 `ack_eliciting_threshold/reordering_threshold/max_ack_delay_ms`
- 新增 HandshakeDelay 帧，字段 `delay_time_us`（4 字节，微秒）
- HandshakeDone 从 1 字节扩展为 9 字节：`type + ack_handshake_pn(8B)`

语义约束：

- Handshake 的 HandshakeDelay 表示“收到 Initial 到发出 Handshake”的处理耗时
- 携带 HandshakeDone 的包也必须携带 HandshakeDelay，表示“收到被确认 Handshake 到发出 HandshakeDone”的处理耗时
- HandshakeDone 指定确认的是哪一个 Handshake 包号，提升重传场景下的收敛精度

版本策略：

- 当前仓库直接升级到协议 v2，不做版本门控（尚未发版，线上无历史协议流量）

### 3.6 拥塞控制与速率控制

当前已落实：

- BbrV1 默认接入发送控制
- Cubic 已接入发送控制（可通过 `cc_algorithm=2` 切换）
- pacer 与发送窗口联合门控
- ACK 驱动带宽与 RTT 采样
- BBR/Cubic 均已支持关键参数配置化（init/min cwnd、增益与阈值）

当前仍未完成：

- BBR 尚有少量关键常量仍为内部固定值（例如 ProbeBW 增益循环、ProbeRTT BDP 倍率、类似 RTT 阈值）
- 拥塞控制专项回归覆盖仍需扩展（模式切换、恢复分支、长时曲线回归）

### 3.7 路径验证、迁移与抗放大

当前已落实：

- 地址变化检测
- PathChallenge / PathResponse
- Unknown / Validating / Validated / Failed 路径状态
- active_path / candidate_path 双路径模型
- validating 阶段的抗放大门禁

当前结论：

- 当前代码已经按“验证成功后再切换”的保守迁移模型工作。
- 候选路径失败不会直接拖垮当前业务路径。

### 3.8 Keepalive 与 MTU

当前已落实：

- Ping 空闲保活
- 收到任意包刷新活动时间
- 连续无响应超时断开
- MTU 梯队探测
- 二分收敛
- MTU 黑洞回退

这意味着当前实现已经具备基础在线路变化场景下的可持续运行能力，而不只是“理论预留”。

### 3.9 安全、票据与 0-RTT

当前已落实：

- X25519Wrapper
- AesGcmContext
- TokenAuth
- Crypto 帧与握手主流程接入
- 包级 AES-GCM 加解密
- SessionToken 编解码与导出
- Connect0RttInfo
- Connect0RttWithStateInfo
- connect0Rtt() / connect0RttWithState()
- 票据验签、有效期校验、peer 绑定校验
- 短窗口抗重放去重

当前接口层也已具备：

- Connection::setOnSessionTokenReady()
- Connection::exportSessionToken()
- Connection::exportSessionResumptionState()

说明：

- 0-RTT 已经不是纯设计草案，而是实际可走通的能力。
- 当前重点已从“是否接入”转向“可观测性、边界补强和回归测试完整性”。

### 3.10 关闭收敛

当前已落实：

- 连接进入 CloseSent/CloseReceived 后停止普通数据面发送
- 再次收到 close 时可在有限次数内重发 close
- 等待若干 PTO 后统一收敛到 Disconnected

关闭行为已经从“收到 close 就立即析构”提升为“有界收尾”。

## 4. 待完善功能

- ProbeRTT 基于 BDP 的倍率参数化
- 类似 RTT 判定阈值参数化

### 4.2 BBR/Cubic 测试覆盖扩展

现状：

- 已有 BBR/Cubic 配置生效与基础行为测试
- 构建与基础回归可稳定通过

缺口：

- BBR 模式迁移（StartUp/Drain/ProbeBW/ProbeRTT）轨迹回归

- BBR 与 Cubic 切换后的对比基准回归

### 4.3 连接级端到端回归

现状：

- 帧级和模块级单测已经较多
- 典型端到端闭环覆盖仍不充分

缺口：

- 主动/被动混合建连回归
- 握手后数据面收发回归（含 HandshakeDone 丢失与恢复路径）
- ACK、丢包、重传串联回归

### 4.4 可观测性仍可继续补强

现状：

缺口：

- 0-RTT 接受/拒绝原因的统计化输出

- MTU 黑洞与恢复路径的统一指标

### 4.5 内部实现一致性

现状：

- 主连接收包路径已经使用 PacketIn 池
- pending 握手解码路径与主路径仍有实现差异

缺口：

- 统一 pending 路径和正式连接路径的对象复用方式
- 减少同一类逻辑在不同路径上的维护分叉

### 4.6 连接级公平调度（WDRR）

现状：

- 已在 Context 写调度中落地 Connection-level WDRR（等权），并提供量子/deficit 上限配置
- 新增连接级调度模式开关：`Disabled` / `Strict` / `WDRR`（默认 `WDRR`）
- stream 内部调度与连接级调度分层工作：连接内先选流，连接间再做轮转

缺口：

- 缺少跨连接公平性回归与可观测统计

### 4.7 非池化内存分配优化

现状：

- 核心报文对象（PacketIn/PacketOut）已实现池化，极大降低了主路径抖动。
- 但在连接管理、控制平面及部分辅助路径上仍存在非池化、高频触发的堆分配。

缺口（优化目标）：

- **Scratch 缓冲区**：`ConnectionImpl` 中的 `m_payloadScratch`、`m_ackPayloadScratch` 等 `std::vector` 在高并发或大帧场景下可能触发 realloc。建议接入 `m_mm` 或采用固定容量的预分配 Buffer。
- **管理容器扩容**：`ContextImpl` 的连接表（`m_connections`）与 `ConnectionImpl` 的流表（`m_streams`）在频繁建连/开流场景下存在节点申请。建议评估静态预分配或自定义池化分配器。
- **CC 内部状态**：BBR 算法中的 `WindowedFilter`（用于带宽和 RTT 采样）存在分散的动态分配。建议将其状态存储与 `ConnectionImpl` 内存块合并分配。
- **丢失历史记录**：`SendControl` 中的 `m_lossSignalsUs` 使用 `std::deque`，高丢包率下频繁插入删除可能导致内存碎片。建议改为固定大小的循环数组。
- **加密上下文**：`AesGcmContext` 与 `X25519Wrapper` 目前在握手阶段通过 `make_shared` 动态创建。对于高并发建连，建议通过对象池（Object Pool）进行复用。
- **辅助函数参数**：`AppendRawBytes` 等函数强依赖 `std::vector` 作为载体。应全面推广指针 + 长度（或 `std::span`）的视图化接口，消除因容器拷贝引起的分配风险。

说明（pending 路径与正式路径的一致性收敛主要指什么）：

- 目前 pending 阶段（被动侧 accept 后、晋升 Connection 前）与正式连接阶段在收包解码、对象池复用、ACK 触发时机、异常路径处理上仍有两套相似但不完全一致的实现。
- 一致性收敛的目标是：同类输入在两条路径上得到尽量一致的行为语义（解码结果、缓存策略、ACK/重传触发、错误码与统计输出）。
- 这项工作本质是“消除双轨实现分叉”，降低维护成本和回归风险，避免同一问题在 pending/正式路径上出现不一致修复。

## 5. 下一步计划

- 扩展连接级公平调度（WDRR）回归与可观测统计，验证多连接长期份额稳定性

### 5.3 P2

- 继续扩展压力测试和长连接回归
- 整理并降低现有工程告警噪声

### 5.4 数据拷贝路径优化清单（后续执行）

以下清单用于指导后续“减少无效拷贝、降低 CPU 与内存带宽开销”的专项优化。

P0（优先落地）：

- 发送路径：减少 Stream 发送缓冲区到组包载荷之间的重复拷贝次数，优先评估批量视图拼装与分片直写可行性（需与 P1 的 scatter/gather 机制联动）。
- 接收路径：优化 Stream 乱序分片重组流程，减少 `RecvFragment -> m_recvBuffer` 的中间复制。
- 加密路径：梳理 `raw_data/encrypt_data` 双缓冲策略，在不影响重传与调试语义的前提下，评估 in-place AEAD 或延迟分配（Encrypt-and-Copy）策略。注意每次重传需分配新 Packet Number 并作为 AAD 参与加密，延迟分配在最终发包前加密更具可行性。

P1（次优先级）：

- 应用读写路径：推广 `acquire/commit` 零拷贝接口使用，减少 `write/read` 传统接口在热点场景下的 memcpy。
- 环形缓冲扩容：优化 `RingBuffer::ensureFree` 触发的整段迁移，评估分级扩容与减少搬迁频次。
- 帧编解码：评估 StreamFrame 载荷从“拷贝编码”向“scatter/gather 发送”演进的边界条件与兼容方案，为 P0 的视图拼装提供底层 `iovec` 数据结构支持。

P2（长期优化）：

- 内核发送路径：在平台允许时评估批量发送 (`sendmmsg`)、`sendmsg + iovec` 以及 GSO（Generic Segmentation Offload）等能力，降低用户态拼包及系统调用拷贝成本。
- 内核接收路径：评估批量收包 (`recvmmsg`)、GRO（Generic Receive Offload）与缓冲复用策略，减少每包动态分配、复制及内核上下文切换开销。
- 异步 I/O 终极演进：评估在 Linux 5.1+ 环境下引入 `io_uring` 替代传统 `libevent` 事件循环，并结合 Fixed Buffers（预注册物理页）实现真正的跨内核边界零拷贝。

验收与约束：

- 逐项优化必须保持现有语义不变：重传、乱序重组、ACK 驱动恢复、加密完整性校验不回退。
- 每项优化需配套微基准与回归：吞吐、P99 时延、CPU 占用、拷贝字节数四项至少覆盖两项。
- 优化默认优先在实现层完成，不改协议线格式；若涉及协议变更，需单独评审。

## 9. 连接级 WDRR 设计草案（待评审）

本节用于评审连接级公平调度方案，仅描述设计与落点，不代表已完成实现。

### 9.1 设计目标

1. 在多连接并发写入时，避免单连接长期占用发送机会。
2. 保持现有拥塞控制（BBR/Cubic）与 stream 调度语义不变。
3. 在不引入协议变更的前提下，完成 Context 层调度公平化。

### 9.2 非目标

1. 本阶段不做差异化权重策略（统一权重）。
2. 本阶段不改动帧格式、ACK 语义、重传语义。
3. 本阶段不引入跨进程或跨实例全局公平。

### 9.3 作用范围

1. 作用层级：Context 层连接调度。
2. 作用对象：已进入可发送路径且存在待发送数据的连接。
3. 不替代：连接内部 stream 调度（Strict/DRR 继续保留）。

### 9.4 核心数据结构

1. active_queue：活跃连接环形队列。
2. deficit_map：`conn -> deficit_bytes`。
3. quantum_map：`conn -> quantum_bytes`（首版统一量子）。
4. in_queue 标记：防止同一连接重复入队。

### 9.5 调度流程

1. 连接调用 wantWrite 时：
将连接加入 active_queue（若未在队列中）。

2. onWriteEvent 主循环：
从队头取连接，累加其 deficit（`deficit += quantum`）。

3. 发送尝试：
若连接可发送且本轮实际发送字节 `sent_bytes > 0`，则 `deficit -= sent_bytes`。

4. 留队规则：
若连接仍有待发送数据且 deficit 仍可支持继续发送，则回到队尾参与下一轮。

5. 退队规则：
若连接无待发送数据或不可发送，则从活跃队列移除，等待下次 wantWrite 再入队。

6. 防饥饿：
每次轮转至少处理一次队头，避免高负载下固定连接长期得不到调度。

### 9.6 与现有模块的关系

1. 与拥塞控制：
WDRR 只决定“谁先发送”，不修改 cwnd/pacing 的计算与约束。

2. 与 stream 调度：
连接内部先由 stream 调度选择帧，连接之间再由 WDRR 分配机会。

3. 与定时器/重传：
重传优先级维持现状；WDRR 仅影响连接轮转次序。

### 9.7 配置建议（首版）

1. `connection_scheduler_mode`：默认 `WDRR`。
2. `connection_wdrr_quantum`：默认 1200 bytes。
3. `connection_wdrr_deficit_cap`：默认 64 KB。

说明：
首版权重固定为 1，后续可扩展为 `quantum = base_quantum * weight`。

### 9.8 可观测性与验收指标

1. per-connection 调度轮次。
2. per-connection 实际发送字节。
3. 队列平均长度与峰值长度。
4. 多连接吞吐份额偏差（目标：长期趋近等权）。
5. 尾时延变化（P95/P99）与重传率变化。

### 9.9 回归测试计划

1. 两连接等负载：长期发送字节比接近 1:1。
2. 多连接不同包长：无连接长期饥饿。
3. 高丢包/高 RTT：不破坏现有重传与拥塞收敛语义。
4. 与 stream DRR 叠加：验证连接间公平和连接内优先级同时生效。

### 9.10 风险与缓解

1. 风险：额外调度状态引入 CPU 开销。
缓解：限制活跃队列操作复杂度，使用 O(1) 入队/出队。

2. 风险：量子设置不当导致吞吐下降或抖动上升。
缓解：提供配置项并给出推荐默认值，保留快速回退能力。

3. 风险：与现有 onWrite 流程耦合导致行为回退。
缓解：先保持单权重最小实现，补齐连接级回归后再扩展权重模型。
