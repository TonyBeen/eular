## 3. 已实现功能

### 3.1 连接与事件模型

当前仓库已经落实以下连接能力：

- Context 提供 bind、connect、accept 等入口。
- 主动建连成功后可进入 Connected。
- 被动建连采用两阶段接入：先收到请求，再由应用显式 accept 放行。
- OnConnected、OnConnectError、OnConnectionClosed 等回调已具备可用语义。
- OnNewConnection 可用于被动连接的准入决策。

当前实现特点：

- 被动连接不是“收到 Initial 就直接变成正式连接”，而是 pending -> accept -> 等待 HandshakeDone -> 晋升 Connected。
- 同步前置失败与异步握手失败都统一汇总到连接结果事件中。

### 3.2 Stream 数据面

当前已落实：

- createStream / getStream
- 流收发缓冲
- read / write
- FIN / RESET_STREAM 基本语义
- 零拷贝 acquire / commit 接口
- 流 ID 准入与活跃流配额校验

说明：

- 当前流模型已经不是“只有帧编解码”，而是完整接入了连接主流程。
- 流关闭、可读、异常终止在 API 层已有对外语义。

### 3.3 Stream 优先级与调度

当前已落实：

- Stream::setPriority() / priority() 接口
- 默认优先级与优先级范围定义
- Config.stream_scheduler_mode
- Config.stream_aging_threshold
- Config.stream_aging_step
- Config.stream_drr_quantum
- Config.stream_drr_deficit_cap
- STRICT 与 DRR 两种调度模式
- 连接级调度统计字段

当前结论：

- stream 优先级能力已经落地，不再属于“只有设计没有实现”的阶段。
- 剩余工作主要是高负载收益验证、指标看板化和参数进一步调优。

### 3.4 包解析与发送对象生命周期

当前已落实：

- 固定头解析
- payload 长度校验
- 帧遍历
- PacketOut 池化管理
- PacketIn 池化接入主收包路径
- PacketOut attempt 链记录每次发送尝试

实现意义：

- 发送侧已经能保留“一个逻辑包多次发送尝试”的轻量历史。
- 接收与发送对象都已有可复用的内存生命周期管理。

### 3.5 ACK、丢包检测与重传

当前已落实：

- Ack 编解码
- ACK 范围解析
- ACK 驱动的 inflight/unacked 回收
- ACK timer
- 包阈值 ACK
- 乱序快速 ACK
- ACK piggyback
- FACK + 重排序阈值丢包检测
- 丢包队列管理
- 使用新包号的重传策略

当前结论：

- 发送恢复闭环已经形成，不再是单向 ACK 解码。
- AckFrequency 不只是帧定义，已经参与 ACK 策略与发送侧重排序阈值联动。

### 3.6 拥塞控制与速率控制

当前已落实：

- BbrV1 默认接入发送控制
- pacer 与发送窗口联合门控
- ACK 驱动带宽与 RTT 采样

当前仍未完成：

- Cubic 尚未形成可切换的完整实现

### 3.7 路径验证、迁移与抗放大

当前已落实：

- 地址变化检测
- PathChallenge / PathResponse
- Unknown / Validating / Validated / Failed 路径状态
- active_path / candidate_path 双路径模型
- validating 阶段的抗放大门禁

当前结论：

- 当前代码已经按“验证成功后再切换”的保守迁移模型工作。
- 候选路径失败不会直接拖垮当前业务路径。

### 3.8 Keepalive 与 MTU

当前已落实：

- Ping 空闲保活
- 收到任意包刷新活动时间
- 连续无响应超时断开
- MTU 梯队探测
- 二分收敛
- MTU 黑洞回退

这意味着当前实现已经具备基础在线路变化场景下的可持续运行能力，而不只是“理论预留”。

### 3.9 安全、票据与 0-RTT

当前已落实：

- X25519Wrapper
- AesGcmContext
- TokenAuth
- Crypto 帧与握手主流程接入
- 包级 AES-GCM 加解密
- SessionToken 编解码与导出
- Connect0RttInfo
- Connect0RttWithStateInfo
- connect0Rtt() / connect0RttWithState()
- 票据验签、有效期校验、peer 绑定校验
- 短窗口抗重放去重

当前接口层也已具备：

- Connection::setOnSessionTokenReady()
- Connection::exportSessionToken()
- Connection::exportSessionResumptionState()

说明：

- 0-RTT 已经不是纯设计草案，而是实际可走通的能力。
- 当前重点已从“是否接入”转向“可观测性、边界补强和回归测试完整性”。

### 3.10 关闭收敛

当前已落实：

- 连接进入 CloseSent/CloseReceived 后停止普通数据面发送
- 再次收到 close 时可在有限次数内重发 close
- 等待若干 PTO 后统一收敛到 Disconnected

关闭行为已经从“收到 close 就立即析构”提升为“有界收尾”。

## 4. 待完善功能

以下能力不是完全没有基础，而是“已有部分实现，但还不够完整或还缺关键收尾”。

### 4.1 Cubic 拥塞控制

现状：

- 头文件与类骨架存在
- 尚未形成完整 Congestion 行为实现
- 当前默认仍以 BBRv1 为主

缺口：

- 算法主体未补齐
- 配置切换后的行为回归尚未建立

### 4.2 连接级端到端回归

现状：

- 帧级和模块级单测已经较多
- 典型端到端闭环覆盖仍不充分

缺口：

- 主动/被动混合建连回归
- 握手后数据面收发回归
- ACK、丢包、重传串联回归
- 路径迁移与抗放大联动回归

### 4.3 可观测性仍可继续补强

现状：

- stream 调度已有一组统计字段
- ACK 与 0-RTT 已有部分行为可观察

缺口：

- 0-RTT 接受/拒绝原因的统计化输出
- path validation 关键事件计数
- MTU 黑洞与恢复路径的统一指标

### 4.4 内部实现一致性

现状：

- 主连接收包路径已经使用 PacketIn 池
- pending 握手解码路径与主路径仍有实现差异

缺口：

- 统一 pending 路径和正式连接路径的对象复用方式
- 减少同一类逻辑在不同路径上的维护分叉

## 5. 下一步计划

下一步建议按优先级推进，而不是平均用力。

### 5.1 P0

- 补齐被动建连与 HandshakeDone 收敛回归测试
- 补齐路径迁移成功/失败/忽略错误响应的行为测试
- 补齐 0-RTT 合法票据、过期票据、无效票据的回归测试
- 增加 0-RTT 接受/拒绝、路径验证结果的可观测统计

### 5.2 P1

- 完善连接级端到端测试场景
- 将 attempt 链统计进一步接入 RTT/重传分析
- 统一 pending 握手路径与主收包路径的池化实现

### 5.3 P2

- 完成 Cubic 并支持稳定切换
- 继续扩展压力测试和长连接回归
- 整理并降低现有工程告警噪声

## 6. 预留接口与未来实现基础

本节只记录“已经存在，但其价值主要体现在未来扩展”的接口、配置或结构。

### 6.1 0-RTT 与会话恢复预留基础

当前已具备的未来扩展基础：

- Connect0RttInfo
- Connect0RttWithStateInfo
- exportSessionToken()
- exportSessionResumptionState()
- OnSessionTokenReady

这些接口说明：

- 会话恢复材料的导出链路已经预留完整。
- 后续若增强票据格式、恢复模式或跨版本兼容，应用侧入口不需要推倒重来。

### 6.2 Stream 调度扩展基础

当前已具备：

- 流优先级字段
- 调度模式枚举
- aging 参数
- DRR 参数
- 连接级调度统计

这些基础意味着后续若继续引入：

- 更细粒度 QoS
- 默认策略热切换
- 按业务类别映射优先级

可以在现有配置和统计框架上继续扩展，而不是重新设计数据面入口。

### 6.3 路径迁移扩展基础

当前已具备：

- 双路径模型
- path_migration_mode 配置入口
- 保守策略已落地
- 激进策略配置位已预留

这说明未来如果要继续实现：

- 更主动的迁移策略
- 更复杂的路径竞争
- PathToken 类路径授权机制

已经有现成的配置入口和状态模型可以承接。

### 6.4 AckFrequency 与恢复策略扩展基础

当前已具备：

- AckFrequency 帧
- 接收侧 ACK 参数应用
- 发送侧重排序阈值联动
- 基于 profile 的自适应更新逻辑

这为后续继续做更智能的 ACK 调优打下了基础，例如按业务模式、链路抖动或丢包特征细化档位策略。

### 6.5 密码学与包级加密基础

当前已具备：

- 握手材料抽象
- AEAD 包级加解密
- 票据完整性保护

这意味着未来如果要增强：

- 加密模式切换
- 更丰富的恢复密钥派生
- 更强的 0-RTT 防重放策略

可以沿现有抽象继续演进。

## 7. 测试与状态结论

当前仓库已经具备较完整的模块级单测，包括：

- 包头与帧解析
- ACK 解码与 ACK 行为
- PathChallenge / PathResponse
- SessionToken
- NetworkPath
- ReceiveHistory
- 0-RTT 与上下文相关能力

总体判断：

- libutp 当前主干能力已经形成闭环，不再处于“只有协议草案”的阶段。
- 后续工作的重点不是从零实现主功能，而是补齐剩余算法、加强回归、增强可观测性，并把已预留能力逐步做实。
