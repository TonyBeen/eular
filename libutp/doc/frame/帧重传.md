# UTP 帧重传策略详解

`UTP_FRAME_RETX_MASK` 定义了**哪些帧类型需要重传**, 哪些不需要, 下面对不需要重传的帧进行解释.

---

## 不重传的帧类型及原因

### 1️⃣ **PADDING (填充帧)**
**原因 1: 无实际数据**
> PADDING 帧没有任何有意义的信息
> - 仅仅是 0x00 字节的序列
> - 不携带任何应用数据或控制信息
> - 重传没有意义

**原因 2: 用途是临时的**
> 典型用途: 填充 Initial 包到 1260 字节 (保证可通过的最小MTU)
> - **MTU 探测**: 填充到特定大小测试路径
> - **Initial 包保护**: 满足最小包大小要求
> - **防止分析**: 混淆流量特征

**原因 3: 丢失无影响**
> 发送: [ACK][STREAM(100字节)][PADDING(50字节)]
> 丢失: 整个包丢失
> 重传: [ACK][STREAM(100字节)]  ← ✅ 不重传 PADDING,节省带宽

---

### 2️⃣ **ACK (确认帧)**

**原因 1: 会自然更新**
```c
// ACK 的生成是动态的
T=0:   收到包 #1, #2, #3
T=1:   发送 ACK{1-3}  (包 #10)
T=10:  包 #10 丢失
T=20:  收到包 #4, #5
T=21:  发送 ACK{1-5}  (包 #11)  ←  包含了之前的信息

// ✅ 包 #11 的 ACK 已经覆盖了包 #10 的 ACK,无需重传
```

**原因 2: ACK 丢失有其他机制处理**
```c
// lsquic_send_ctl.c
if (packet_out->po_frame_types & (1 << QUIC_FRAME_ACK)) {
    ctl->sc_flags |= SC_LOST_ACK_INIT << pns;  //  标记 ACK 丢失
}

// 处理 ACK 丢失 (lsquic_conn.c)
if (send_ctl->sc_flags & SC_LOST_ACK_APP) {
    generate_ack_frame(conn);  //  生成新的 ACK 帧,而不是重传旧的
}
```

**原因 3: ACK-only 包的特殊处理**
```c
// RFC 9000, Section 13.2.4:
// "Packets containing only ACK frames are not congestion controlled"

// 丢失的 ACK-only 包不计入拥塞控制
if (packet_out->po_frame_types == QUIC_FTBIT_ACK) {
    // ❌ 不触发拥塞控制,也不重传
    return;
}
```

---

### 3️⃣ **PING (心跳帧)**
**原因 1: 会定期发送**
```c
// lsquic_conn.c - 心跳逻辑
static void send_keepalive (struct lsquic_conn *lconn) {
    lsquic_alarmset_set(&lconn->cn_alset, AL_PING, 
                        now + PING_PERIOD);  //  定期触发

    packet_out = get_packet(lconn);
    lsquic_packet_out_add_ping_frame(packet_out);
}

// ✅ 下一个 PING 会自然发送,无需重传旧的
```

**原因 2: 目的是触发 ACK**
```c
// PING 的作用: 探测连接存活
T=0:   发送 PING  (包 #100)
T=50:  包 #100 丢失  ← ❌ 对端没收到,不会回 ACK
T=60:  发送 PING  (包 #101, 定期心跳)
T=61:  收到 ACK{#101}  ← ✅ 连接正常

// 重传包 #100 的 PING 没有意义,因为包 #101 已经达到目的
```

**原因 3: 避免重复探测**
```c
// 场景: PING 丢失后的处理
if (no_ack_received_for_long_time) {
    send_new_ping();  //  发送新的 PING
    // ❌ 不重传旧的 PING,因为:
    // 1. 如果网络恢复,新 PING 足够
    // 2. 如果网络断开,重传也没用
}
```

**原因 4: 影响拥塞控制**
```c
// RFC 9000, Section 19.2:
// "A packet containing only a PING frame is not congestion controlled"

// PING-only 包的特殊处理
if (packet_out->po_frame_types == QUIC_FTBIT_PING) {
    // ✅ 不计入 in-flight
    // ✅ 丢失不触发拥塞控制
    // ❌ 但也因此不重传
}
```
