# 连接与异常处理流程

> 更新时间：2026-03-22  
> 范围：当前 `ContextImpl` / `ConnectionImpl` 已实现行为（代码语义对齐）

---

## 1. 连接流程（主动）

1. 应用调用 `ContextImpl::connect(info)`。
2. 参数校验：
   - `info.ip` 为空或 `info.port==0`，直接返回错误。
   - UDP socket 未 `bind`，直接返回错误。
3. 重复连接检查：
   - 若同目标地址已有连接且处于 pending，返回 `UTP_ERR_IN_PROGRESS`。
   - 若同目标地址已连接成功，返回 `UTP_ERR_SOCKET_CONNECTED`。
4. 分配本地 CID，创建 `ConnectionImpl`，进入 `kStateWaitSendInitial`。
5. 写事件中发送 Initial：
   - 成功后进入 `kStateInitialSent`。
   - 失败则记录 `send initial packet failed`，状态转 `kStateDisconnected`。
6. 收到对端 Handshake 后进入 `kStateConnected`，并启动 HandshakeDone 发送逻辑。

## 2. 连接失败处理（主动与通用）

补充说明（当前实现）：

1. 同步前置失败（如参数非法、未 bind、重复连接、0-RTT state 前置校验失败）会在函数返回错误前触发 `OnConnectError`。
2. 异步握手失败（如超时、握手期被关闭、被动提升冲突）同样触发 `OnConnectError`。
3. `OnConnectError` 的第三个参数统一为 `ConnectAttemptInfo`，可同时表达普通握手、0-RTT(ticket)、0-RTT(state)、被动建连。

`ConnectAttemptInfo` 字段语义（关键项）：

| 字段 | 说明 |
|---|---|
| `type` | 尝试类型：`Normal / ZeroRttToken / ZeroRttState / Passive` |
| `session_token_size` | 仅在 `ZeroRttToken` 有意义 |
| `resumption_state_size` | 仅在 `ZeroRttState` 有意义 |
| `early_data_size` | 本次尝试携带的早数据长度 |
| `early_fin` | 是否请求早数据结束时发送 FIN |

## 2.1 典型失败来源

- 建连参数非法：`UTP_ERR_INVALID_PARAM`
- 未绑定 socket：`UTP_ERR_SOCKET_NOT_BOUND`
- 连接已存在/重复：`UTP_ERR_IN_PROGRESS`、`UTP_ERR_SOCKET_CONNECTED`
- Initial 发送失败：记录 `send initial packet failed` 并断开
- 连接超时：`onConnTimeout()` 设置 `UTP_ERR_TIMEOUT` + `connect timeout`
- 路径验证超时失败：`onPathValidationTimeout()` 设置 `UTP_ERR_TIMEOUT` + `path validation timeout`
- 握手期间收到关闭：进入关闭态，最终按 pending 失败路径收敛

## 2.2 回调收敛规则（`ContextImpl::handleConnectionState`）

- 若连接还在 pending 集合中（尚未建立）且状态变为：
  - `kStateCloseSent` / `kStateCloseReceived` / `kStatePtoTimedWait` / `kStateDisconnected`
   - 走 `OnConnectError(errorCode, reason, connectAttemptInfo)`
- 若连接已建立后转 `kStateDisconnected`：
  - 走 `OnConnectionClosed(connection)`
- 最后从 `m_connections` 中移除连接对象。

这保证了“建连失败”和“已连接后断开”的回调语义分离。

## 2.3 同步与异步错误的一致性约束

1. 对调用方而言，`OnConnectError` 是统一错误观测入口，不需要再按“返回值错误 vs 握手阶段错误”分两套处理。
2. 由于同步失败也会触发回调，应用可在回调中按 `ConnectAttemptInfo.type` 做统一打点与重试策略。
3. 若应用不希望同步失败触发重复处理，可在调用点用 request-id 去重（例如按 ip/port/type 和时间窗口去重）。

## 2.4 实际事件时序示例

### 示例 A：普通握手参数错误（同步失败）

1. 应用调用 `connect(ConnectInfo{ip="", port=0, ...})`
2. 内部参数校验失败，设置 `UTP_ERR_INVALID_PARAM`
3. 触发 `OnConnectError(code, reason, attemptInfo)`
   - `attemptInfo.type = Normal`
   - `attemptInfo.session_token_size = 0`
   - `attemptInfo.resumption_state_size = 0`
4. `connect(...)` 返回错误（`-1`）

说明：该场景没有进入网络握手阶段，但仍统一走 `OnConnectError`。

### 示例 B：0-RTT Token 直连（异步失败）

1. 应用调用 `connect0Rtt(Connect0RttInfo{session_ticket=..., early_data=...})`
2. 本地前置校验通过并返回 `UTP_ERR_OK`
3. 进入异步握手阶段，后续发生超时/握手期关闭
4. 连接状态从 pending 收敛到 failed，触发 `OnConnectError(...)`
   - `attemptInfo.type = ZeroRttToken`
   - `attemptInfo.session_token_size > 0`
   - `attemptInfo.early_data_size` 为本次早数据长度

说明：该场景是“调用成功返回 + 后续异步失败”的典型路径。

### 示例 C：0-RTT State 恢复串非法（同步失败）

1. 应用调用 `connect0RttWithState(info, state)`
2. 本地前置校验失败（如 Base64 非法/AEAD 解密失败/本地过期）
3. 触发 `OnConnectError(code, reason, attemptInfo)`
   - `attemptInfo.type = ZeroRttState`
   - `attemptInfo.resumption_state_size = state.size()`
4. `connect0RttWithState(...)` 返回错误（`-1`）

说明：应用可不区分失败来源，统一在 `OnConnectError` 中按 `type` 做路由处理。

## 3. 中途创建 Stream 与超限处理

入口是 `ConnectionImpl::createStream()`。

1. 状态检查：
   - 非 `kStateConnected` 直接返回 `UTP_ERR_INVALID_STATE`。
2. 对端流控上限检查：
   - 上限取 `max(1, m_peerTP.init_max_streams_bidi)`。
   - 若当前 `m_streams.size() >= maxStreams`，返回 `UTP_ERR_STREAM_LIMIT_ERROR`。
3. Stream ID 分配：
   - 主动发起方从 0 起步，被动侧从 1 起步。
   - 每次 `+4` 保持分片空间。
   - 若发生 ID 冲突，返回 `UTP_ERR_STREAM_ID_EXHAUSTED`。
4. 创建成功后：
   - 触发 `setOnIncomingStream()` 注册的回调。
   - 若仍可继续创建（`m_streams.size() < maxStreams`），触发 `OnStreamCanCreate()`。

说明：当前超限策略是“立即拒绝创建并返回错误码”，不做排队等待。

## 4. 断连处理（ConnectionClose / PTO 收尾）

## 4.1 本地主动关闭

1. 应用调用 `ConnectionImpl::close()`：
   - 状态置为 `kStateCloseSent`
   - 停止连接相关计时器
   - 设置关闭原因（默认 `UTP_ERR_CANCELLED` / `local close`）
   - 标记 `m_closeFramePending=true`
2. 写事件 `onWrite()` 中发送 ConnectionClose：
   - 成功：进入 `enterPtoTimedWait()`
   - `UTP_ERR_WOULD_BLOCK`：重新 `scheduleWrite()`
   - 其他失败：直接转 `kStateDisconnected`

## 4.2 收到对端 ConnectionClose

1. 解析帧时识别到 `ConnectionClose`：
   - 状态设为 `kStateCloseReceived`
   - 记录关闭错误信息
   - 设置 `m_closeFramePending=true`，准备回发 close
2. 写事件发送本端 close 成功后进入 PTO timed-wait。
3. 一旦进入 `kStateCloseSent` 或 `kStateCloseReceived`：
   - 停止正常 stream 数据面发送。
   - 接收侧只保留 `ConnectionClose` 的最小处理，其它业务帧直接忽略。
   - 此时应用继续写 stream，会返回 `UTP_ERR_CONNECTION_CLOSING`，用于明确区分“关闭中”与普通 `UTP_ERR_INVALID_STATE`。

## 4.3 PTO 收尾窗口与被动重发

- `enterPtoTimedWait()` 会计算 PTO，并设置截止时间 `deadline = now + 3 * PTO`。
- 在 `kStatePtoTimedWait` 下：
  - 不主动定时重发 close。
  - 仅当收到对端 `ConnectionClose` 帧时，才尝试回发 close。
  - 回发限频：
    - 最多 3 次（计数 `m_closePeerResendCount`）。
    - 且满足最小间隔保护（`max(PTO/2, 1ms)`）。
- 到达 3*PTO 超时：
  - `onCloseDrainTimeout()` 将状态转 `kStateDisconnected` 并调度状态收敛。

---

## 5. 回调语义速查

| 场景 | 状态收敛 | 回调 |
|---|---|---|
| 建连成功 | pending -> connected | `OnConnected` |
| 建连阶段失败/被关闭 | pending -> close/disconnected | `OnConnectError` |
| 已连接后正常或异常断开 | connected -> disconnected | `OnConnectionClosed` |

---

## 6. 设计取舍说明

- 关闭阶段采用“3*PTO 观察窗口 + 被动重发”策略，避免无条件主动重发带来的额外噪声。
- Stream 超限采用 fail-fast，便于上层尽早感知流控压力并做重试/退避策略。
- 通过 `handleConnectionState()` 统一回调出口，减少连接生命周期分叉导致的状态不一致。

## 7. Keepalive（空闲保活）

当前实现已支持连接级保活，行为如下：

1. 连接进入 `kStateConnected` 后启动 keepalive 定时器。
2. 收到任意入站包会刷新活动时间，并清零“连续无响应探测计数”。
3. 当连接空闲达到 keepalive 间隔后，发送 `Ping` 探测包。
4. 每次探测后在 `keepalive_timeout` 时间内未收到任何包，计数加 1。
5. 当连续无响应次数达到 `keepalive_probes`（N，可配置）后，连接以 timeout 原因断开。

配置项：

- `enable_keepalive`：开关
- `keepalive_interval`：空闲探测间隔（ms）
- `keepalive_timeout`：单次探测等待响应超时（ms）
- `keepalive_probes`：连续无响应最大次数 N
