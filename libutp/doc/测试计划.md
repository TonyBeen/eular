## 阶段 3：网络仿真测试（netem）

**目标**：在模拟真实网络条件下验证传输正确性与拥塞控制收敛  
**依赖**：Linux `tc qdisc`（iproute2）、echo_server/echo_client example  
**场景矩阵**：

| 场景 | netem 参数 | 验收指标 |
|------|-----------|---------|
| 基准（无损） | 无 | 吞吐 ≥ 基线 90% |
| 低丢包 | loss 1% | 连接建立成功，吞吐 ≥ 基线 70% |
| 高丢包 | loss 5% | 连接建立成功，无死锁 |
| 延迟 | delay 50ms | RTT 测量接近 50ms，无超时断连 |
| 高延迟 | delay 200ms | 握手成功（超时窗口内） |
| 乱序 | reorder 10% | 数据完整送达，无 SEGFAULT |
| 抖动 | delay 20ms jitter 10ms | 传输完成 |
| 综合 | loss 1% delay 30ms reorder 5% | 传输完成 |

当前建议执行顺序：

1. 基础场景与旧回归：
- `test/scripts/netem_test.sh`

2. 增强场景与主要验收入口：
- `test/scripts/netem_enhanced_test.sh`

增强场景覆盖：

- 100MB 级大流量传输
- Gemodel 突发丢包（轻度 / 中度 / 重度）
- 10 并发总量 100MB
- 当前主线版本应至少满足 `PASS 15 / FAIL 0`

补充说明：

- `netem_enhanced_test.sh` 已经比原始 `netem_test.sh` 更贴近当前仓库的真实验收需求。
- 后续若只保留一个主脚本，优先保留增强版，再把基础版场景并入增强版。

## 阶段 4：协议正确性与状态机回归

**目标**：确保协议状态迁移、包格式、关键控制帧和尾部收敛在代码演进后持续正确。  
**当前资产**：

- `test_frame_*.cc`
- `test_ack_behavior.cc`
- `test_context_integration.cc`
- `test_connection_stream_layer.cc`
- `test_context_zero_rtt.cc`

重点关注：

- Initial / Handshake / HandshakeDone
- ACK / AckFrequency / HandshakeDelay
- 纯 FIN、流 EOF、RESET_STREAM
- 路径验证与地址变化
- 0-RTT 接受 / 拒绝 / 重放防护

当前要求：

- 任何握手、ACK、流关闭、路径迁移改动，都应至少补充一条对应单测或集成回归。

## 阶段 6：长稳压力测试

**目标**：验证 6+ 小时持续运行下无内存泄漏、fd 泄漏  
**场景**：连接循环建立/关闭 + 大小包混合 + 多流并发  
**监控**：/proc/PID/smaps、fd 数、Valgrind massif

建议补充：

- ASan/UBSan 长时间运行
- 定时建立/关闭连接
- 小包、大包、0-RTT、路径变化混合流量
- 定期采样 RSS、fd 数、连接数、stream 数和重传率

验收重点：

- 无持续性 RSS 漏涨
- 无 fd 漏涨
- 无连接表 / stream 表遗留
- 无长时间运行后握手成功率明显劣化

---

## 阶段 7：性能基线测试

**目标**：建立吞吐/延迟/CPU 基线，防止性能回归  
**工具**：example echo_client/server + iperf-like 测量  
**输出**：P50/P99 往返延迟、吞吐 (MB/s)、CPU 占用率

建议固定基线维度：

- 单连接小包时延
- 单连接大流吞吐
- 1% 丢包下吞吐保持率
- 10 并发总吞吐
- 建连成功率与握手耗时分位数

建议输出：

- P50 / P95 / P99 RTT
- 吞吐 MB/s
- CPU 占用
- 重传率
- `connect timeout` 比例

---

## 阶段 8：高并发与极限压力测试

**目标**：验证连接级调度、公平性、内存占用和定时器/事件分发在高并发下不失控。  
**建议场景**：

- 大量短连接循环建立/关闭
- 多连接多流并发
- 大小包混合
- 丢包、延迟和乱序叠加

重点观察：

- CPU 与事件循环开销
- 连接调度公平性
- ACK 帧膨胀
- 内存碎片与对象池增长
- 重传队列与 timer 压力

---

## 阶段 9：安全性与鲁棒性补充

**目标**：验证异常输入、恶意包和边界条件下不会崩溃或出现资源放大。  
**建议方向**：

- 非法帧长度 / 非法参数组合
- 重复包 / 乱序极端场景
- 票据伪造 / 过期 / 重放
- 握手尾部迟到包
- MTU 黑洞与反复路径变化

## 执行顺序

```
阶段1 → 阶段2 → 阶段3 → 阶段4 → 阶段5 → 阶段6 → 阶段7 → 阶段8 → 阶段9
 基线      注入      netem    协议回归   Fuzz     长稳      性能      并发      安全
```

当前优先级建议：

1. 日常回归以 `utp_tests + netem_enhanced_test.sh` 为主。
2. 协议主路径变更时，优先补阶段 4 的状态机回归。
3. 发版前补跑阶段 5、阶段 7、阶段 8。
4. 安全和 Fuzz 适合持续集成或周期性专项执行。
